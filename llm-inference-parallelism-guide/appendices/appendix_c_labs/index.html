    부록 C. 실습 목록¶

    이 부록은 본문 개념을 직접 확인하기 위한 실습 아이디어다. 모든 실습을 한 번에 할 필요는 없다. 작은 모델과 짧은 prompt로 시작하고, token 수와 latency를 기록하는 습관을 들이는 것이 중요하다.

    C.1 단일 GPU vLLM 실행¶

    C.1.1 서버 실행¶

    목표: OpenAI-compatible LLM server를 직접 띄운다.

    예시:

    vllm serve <MODEL>

    확인할 것:

    • 서버가 정상적으로 model을 load하는가?
    • GPU memory 사용량은 얼마인가?
    • 기본 endpoint가 응답하는가?

    C.1.2 Python client 호출¶

    목표: OpenAI-compatible client로 self-hosted model을 호출한다.

    확인할 것:

    • base_url을 로컬 server로 바꾸어 호출할 수 있는가?
    • max_tokens를 바꾸면 output 길이와 latency가 어떻게 달라지는가?
    • Temperature를 낮추면 재현성이 좋아지는가?

    C.1.3 Streaming 응답 확인¶

    목표: TTFT와 TPOT를 체감한다.

    확인할 것:

    • 첫 token이 나오기까지 얼마나 걸리는가?
    • Token이 일정하게 이어지는가?
    • Prompt 길이를 늘리면 TTFT가 어떻게 달라지는가?

    C.2 성능 측정¶

    C.2.1 TTFT 측정¶

    목표: Prompt 길이와 TTFT의 관계를 측정한다.

    방법:

    1. 짧은 prompt, 중간 prompt, 긴 prompt를 준비한다.
    2. Streaming 요청을 보낸다.
    3. 요청 시작 시각과 첫 token 수신 시각의 차이를 기록한다.

    기대 관찰:

    • Prompt가 길수록 TTFT가 증가한다.

    기록 양식:

    실험 Prompt tokens Output tokens TTFT(ms) End-to-end latency(ms) 메모
    짧은 prompt
    중간 prompt
    긴 prompt

    C.2.2 TPOT 측정¶

    목표: Decode 속도를 측정한다.

    방법:

    1. 첫 token 이후 각 token 수신 시간을 기록한다.
    2. Output token 수와 decode 시간을 나누어 평균 TPOT를 계산한다.

    기대 관찰:

    • Context가 길거나 batch가 커지면 TPOT가 달라질 수 있다.

    계산식:

    TPOT = 첫 token 이후의 decode 시간 / 첫 token 이후 생성된 output token 수

    C.2.3 Batch size별 throughput 비교¶

    목표: Batch size와 throughput/latency trade-off를 본다.

    방법:

    1. 동시 요청 수를 1, 2, 4, 8로 늘린다.
    2. Tokens/sec, requests/sec, p95 latency를 기록한다.

    기대 관찰:

    • 동시 요청이 늘면 throughput은 좋아질 수 있지만 latency가 나빠질 수 있다.

    기록 양식:

    동시 요청 수 Requests/sec Tokens/sec p95 latency(ms)
    1
    2
    4
    8

    추가로 GPU memory peak와 OOM 발생 여부를 메모한다.

    C.3 병렬화 실습¶

    C.3.1 Tensor parallel 실행¶

    목표: tensor_parallel_size의 의미를 확인한다.

    예시:

    vllm serve <MODEL> \
  --tensor-parallel-size 2

    확인할 것:

    • 두 GPU가 모두 사용되는가?
    • 단일 GPU 실행과 memory 사용량이 어떻게 다른가?
    • Latency와 throughput은 어떻게 달라지는가?

    C.3.2 Pipeline parallel 실행¶

    목표: TP와 PP 조합을 개념적으로 확인한다.

    예시:

    vllm serve <MODEL> \
  --tensor-parallel-size 2 \
  --pipeline-parallel-size 2

    확인할 것:

    • 필요한 총 GPU 수를 계산할 수 있는가?
    • 모델과 GPU 구성에 맞는 설정인가?

    C.3.3 Long-context 요청 실험¶

    목표: Context length가 TTFT, TPOT, GPU memory에 미치는 영향을 본다.

    방법:

    1. 1K, 4K, 16K 이상의 prompt를 준비한다.
    2. 같은 output length로 요청한다.
    3. TTFT, TPOT, memory usage를 기록한다.

    기대 관찰:

    • Prompt가 길수록 TTFT가 증가한다.
    • Context가 길수록 KV cache memory가 증가한다.

    C.4 운영 전 점검표¶

    실습을 마친 뒤에는 다음 항목을 간단히라도 기록해 둔다. 이 표가 있어야 나중에 모델, GPU, 엔진 버전을 바꾸었을 때 성능 변화를 비교할 수 있다.

    항목 확인 내용
    모델 모델 이름, parameter 크기, quantization 여부
    GPU GPU 종류, 개수, GPU memory, interconnect
    엔진 vLLM/SGLang/TensorRT-LLM 버전과 주요 실행 옵션
    요청 패턴 평균 prompt tokens, 평균 output tokens, 동시 요청 수
    Latency TTFT, TPOT, end-to-end latency의 p50/p95
    Throughput requests/sec, input tokens/sec, output tokens/sec
    Memory 초기 weight load 후 memory, peak memory, OOM 발생 조건
    품질 temperature, top-p, structured output 사용 여부
