    29장. Speculative Decoding¶

    Speculative decoding은 decode 병목을 줄이기 위한 기법이다. 작은 draft model이 후보 token을 여러 개 먼저 제안하고, 큰 target model이 이를 검증한다. 검증을 통과한 token은 한 번에 받아들여 decode step 수를 줄일 수 있다.

    학습 목표¶

    • Speculative decoding의 기본 아이디어를 이해한다.
    • Draft model과 target model의 역할을 구분한다.
    • 병렬화 및 scheduling과의 상호작용을 파악한다.

    29.1 기본 아이디어¶

    Autoregressive decode는 token을 하나씩 생성한다. Speculative decoding은 이 순차성을 완전히 없애지는 못하지만, 여러 후보 token을 미리 제안하고 검증해 일부 step을 줄인다.

    29.1.1 작은 draft model이 후보 token을 생성한다¶

    Draft model은 target model보다 작고 빠른 모델이다. Draft model이 다음 몇 개 token을 빠르게 생성한다. 이 후보들은 target model이 만들 가능성이 높은 token이라고 기대한다.

    29.1.2 큰 target model이 검증한다¶

    Target model은 실제 품질 기준이 되는 큰 모델이다. Draft model이 제안한 token들을 target model이 검증한다. 검증을 통과한 token은 받아들이고, 틀린 지점부터 다시 target model 기준으로 진행한다.

    29.1.3 맞는 token은 한 번에 여러 개 받아들인다¶

    Draft model이 target model과 비슷한 token을 잘 제안하면, target model은 여러 token을 한 번에 받아들일 수 있다. 그러면 decode loop의 effective step 수가 줄어든다.

    반대로 draft model 제안이 자주 틀리면 이득이 작아진다.

    29.2 장점과 한계¶

    Speculative decoding은 decode 병목을 완화할 수 있지만, 모든 workload에 항상 효과적인 것은 아니다.

    29.2.1 Decode 속도를 높일 수 있다¶

    Target model이 매 token을 하나씩 생성하는 대신, draft 후보를 검증하며 여러 token을 받아들이면 TPOT가 개선될 수 있다. 특히 target model이 크고 decode가 병목일 때 매력적이다.

    29.2.2 Draft model 품질이 중요하다¶

    Draft model이 target model의 분포를 잘 따라가야 한다. 너무 작은 모델이 부정확한 후보를 많이 만들면 검증 실패가 많아져 이득이 줄어든다.

    Draft model 자체를 실행하는 비용도 있으므로, 품질과 속도의 균형이 중요하다.

    29.2.3 모든 workload에서 효과적인 것은 아니다¶

    Structured output, constrained decoding, 매우 짧은 output, 복잡한 sampling 설정에서는 효과가 제한적일 수 있다. Batch scheduling과도 상호작용한다.

    따라서 speculative decoding은 benchmark와 실제 workload 기준으로 측정해야 한다.

    29.3 병렬화와의 관계¶

    Speculative decoding은 병렬화와 별개의 기법이지만, serving 시스템 안에서는 함께 고려된다.

    29.3.1 Decode 병목 완화¶

    Speculative decoding의 직접 목표는 decode 병목 완화다. Prefill이 병목인 긴 prompt workload에서는 효과가 제한적일 수 있다. 먼저 병목이 TTFT인지 TPOT인지 확인해야 한다.

    29.3.2 Batch scheduling과 상호작용¶

    Draft와 target 실행이 추가되므로 scheduler가 더 복잡해질 수 있다. 어떤 요청은 여러 token이 한 번에 accept되고, 어떤 요청은 reject되어 다시 진행한다. Batch 안의 요청 진행 속도가 달라질 수 있다.

    29.3.3 Production 적용 난이도¶

    Production에서는 draft model 선택, memory overhead, 품질 검증, rollback, monitoring이 필요하다. 단순 옵션 하나로 끝나는 문제가 아닐 수 있다.

    정리¶

    Speculative decoding은 작은 draft model이 후보 token을 제안하고 큰 target model이 검증해 decode step을 줄이는 기법이다. Decode 병목을 완화할 수 있지만, draft model 품질과 workload 특성에 따라 효과가 달라진다.
