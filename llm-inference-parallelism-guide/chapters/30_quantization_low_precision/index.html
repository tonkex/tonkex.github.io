    30장. Quantization과 저정밀 추론¶

    Quantization은 LLM 추론 비용을 줄이는 가장 현실적인 도구 중 하나다. 모델 weight를 더 낮은 precision으로 표현하면 GPU memory 사용량이 줄고, memory bandwidth 부담도 줄어들 수 있다. 하지만 정확도 저하 가능성이 있으므로 품질 평가가 반드시 필요하다.

    학습 목표¶

    • FP16, BF16, FP8, INT8, INT4의 큰 차이를 이해한다.
    • Quantization이 memory와 latency에 미치는 효과를 설명한다.
    • 개발자 관점에서 품질 평가가 필요한 이유를 정리한다.

    30.1 Precision의 종류¶

    Precision은 숫자를 표현하는 방식이다. LLM weight와 activation은 보통 floating point나 integer 형식으로 저장되고 계산된다. Precision이 낮아지면 memory는 줄지만 표현 가능한 값의 세밀함도 줄어든다.

    30.1.1 FP16과 BF16¶

    FP16과 BF16은 LLM 추론에서 널리 쓰이는 16-bit floating point 형식이다. FP32보다 memory를 절반으로 줄이고 GPU tensor core를 잘 활용할 수 있다. BF16은 FP16보다 exponent 범위가 넓어 안정성이 좋은 경우가 많다.

    30.1.2 FP8¶

    FP8은 8-bit floating point 형식이다. 최신 GPU에서 지원이 강화되고 있으며, memory와 bandwidth를 더 줄일 수 있다. 하지만 모델과 hardware, framework 지원을 확인해야 한다.

    30.1.3 INT8과 INT4¶

    INT8과 INT4는 integer 기반 quantization이다. INT4는 weight memory를 크게 줄일 수 있어 로컬 추론과 큰 모델 serving에서 자주 사용된다. 다만 quantization 방식에 따라 품질 차이가 크다.

    30.2 Quantization의 효과¶

    Quantization의 가장 직접적인 효과는 model weight memory 감소다. 경우에 따라 latency와 throughput도 개선될 수 있다.

    30.2.1 Model weight 메모리를 줄인다¶

    FP16 weight는 parameter 하나에 2 byte를 사용한다. INT4 weight는 이상적으로 훨씬 적은 memory를 사용한다. 큰 모델이 GPU에 들어가지 않을 때 quantization은 첫 번째로 검토할 전략이다.

    30.2.2 더 큰 모델을 올릴 수 있다¶

    Weight memory가 줄면 같은 GPU에 더 큰 모델을 올릴 수 있다. 또는 남는 memory를 KV cache에 사용할 수 있어 더 긴 context나 더 큰 batch를 처리할 수 있다.

    하지만 quantization이 KV cache 자체를 자동으로 줄이는 것은 아니다. KV cache quantization은 별도 주제다.

    30.2.3 정확도 저하 가능성이 있다¶

    낮은 precision은 모델 출력을 바꿀 수 있다. 일반 대화에서는 문제가 작아 보여도, 코드 생성, 수학, 도메인 특화 QA에서는 품질 저하가 드러날 수 있다.

    따라서 quantized model을 적용할 때는 실제 workload 평가셋으로 품질을 비교해야 한다.

    30.3 개발자 관점의 선택¶

    개발자는 직접 quantization 알고리즘을 구현하기보다, 검증된 quantized model과 framework 지원을 활용하는 경우가 많다.

    30.3.1 먼저 검증된 quantized model을 사용한다¶

    이미 많이 사용되고 검증된 quantized model을 선택하면 시행착오를 줄일 수 있다. 모델 카드, benchmark, community report, framework compatibility를 확인한다.

    30.3.2 품질 평가를 반드시 수행한다¶

    서비스의 실제 prompt와 expected behavior를 기준으로 평가한다. 단순 benchmark 점수만으로 충분하지 않다. 답변 정확도, format 준수, hallucination, latency, memory를 함께 비교한다.

    30.3.3 비용과 latency를 함께 본다¶

    Quantization은 memory를 줄이지만 항상 latency를 줄이는 것은 아니다. Kernel 지원이 약하거나 dequantization overhead가 있으면 기대만큼 빨라지지 않을 수 있다. 실제 serving 환경에서 측정해야 한다.

    정리¶

    Quantization은 model weight memory를 줄이고 더 큰 모델을 실행 가능하게 만드는 강력한 도구다. FP16/BF16, FP8, INT8, INT4는 memory와 품질의 trade-off가 다르다. 개발자는 검증된 quantized model로 시작하되, 반드시 실제 workload 기준 품질과 latency를 측정해야 한다.
