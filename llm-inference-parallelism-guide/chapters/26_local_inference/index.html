    26장. 개인 개발용 로컬 추론¶

    개인 개발용 로컬 추론은 production serving과 목표가 다르다. 최대 throughput보다 설치와 사용 편의성, 반복 실험, privacy, offline 사용성이 더 중요할 수 있다. 작은 모델과 quantization을 활용하면 노트북이나 단일 GPU에서도 충분히 유용한 실습 환경을 만들 수 있다.

    학습 목표¶

    • 단일 GPU 또는 Apple Silicon 환경에서의 현실적인 모델 선택 기준을 이해한다.
    • 로컬 RAG 실습에서 prompt 길이 관리가 중요한 이유를 파악한다.
    • 로컬 agent 실습에서 로그와 재현성을 관리한다.

    26.1 단일 GPU 또는 Apple Silicon 환경¶

    로컬 환경에서는 hardware 제약을 먼저 받아들여야 한다. GPU memory, unified memory, CPU 성능, disk 공간이 모두 제한적이다. 따라서 모델 크기와 context length를 현실적으로 선택하는 것이 중요하다.

    26.1.1 작은 모델 선택¶

    처음에는 1B-8B급 instruct model처럼 작은 모델로 시작하는 것이 좋다. 작은 모델은 품질 한계가 있지만, 빠르게 실험하고 시스템 구조를 이해하는 데 유리하다.

    큰 모델을 억지로 올리는 것보다 작은 모델로 prompt, RAG, agent loop를 안정적으로 만드는 것이 학습에 더 도움이 된다.

    26.1.2 Quantized model 활용¶

    Quantized model은 로컬 추론에서 특히 유용하다. INT4나 GGUF 계열 모델을 사용하면 제한된 memory에서도 실행 가능성이 높아진다. 다만 quantization 방식에 따라 품질과 속도가 달라지므로, 실제 task 기준으로 비교해야 한다.

    26.1.3 Latency보다 사용 편의성을 우선한다¶

    개인 실습에서는 production 수준의 p95 latency보다 개발자가 쉽게 실행하고 디버깅할 수 있는지가 중요하다. 모델 로딩, prompt 변경, 로그 확인, 결과 재현이 쉬운 환경을 먼저 만든다.

    26.2 로컬 RAG 실습¶

    RAG는 로컬 추론과 잘 어울리는 실습 주제다. 문서를 chunk로 나누고, embedding을 만들고, vector DB에서 검색한 뒤 LLM prompt에 넣는 흐름을 직접 경험할 수 있다.

    26.2.1 문서 chunking¶

    문서를 너무 크게 chunk하면 검색 결과 하나가 많은 token을 차지한다. 너무 작게 chunk하면 문맥이 끊길 수 있다. 입문자는 chunk size와 overlap을 바꿔 보며 prompt token 수와 답변 품질 변화를 관찰하는 것이 좋다.

    26.2.2 Embedding과 vector DB¶

    Embedding model과 vector DB는 RAG의 검색 품질을 좌우한다. 로컬에서는 가벼운 embedding model과 간단한 vector store로 시작할 수 있다. 검색 품질이 낮으면 LLM context에 관련 없는 문서가 들어가 비용만 증가한다.

    26.2.3 Prompt 길이 관리¶

    검색된 문서를 많이 넣으면 답변 품질이 항상 좋아지는 것은 아니다. Prompt가 길어져 TTFT가 증가하고 KV cache가 커진다. 로컬 실습에서도 token count를 출력하고, top-k 문서 수를 바꾸며 latency를 관찰한다.

    26.3 로컬 agent 실습¶

    Agent 실습에서는 tool calling, multi-step reasoning, memory 관리가 중요하다. 로컬 환경에서는 실행 로그를 자세히 남기는 것이 특히 유용하다.

    26.3.1 Tool calling¶

    Tool calling은 LLM이 외부 함수나 API를 호출하도록 하는 패턴이다. 로컬에서는 파일 검색, 계산기, 간단한 HTTP 요청처럼 위험이 낮은 도구부터 시작한다. Tool schema가 길어지면 prompt token 수가 늘어난다는 점도 확인한다.

    26.3.2 Multi-step reasoning¶

    Agent는 여러 단계로 생각하고 실행한다. 단계가 늘어날수록 context도 길어진다. 중간 결과를 모두 prompt에 유지하면 비용이 빠르게 늘 수 있다.

    필요한 정보만 남기고 오래된 중간 로그는 요약하거나 제거하는 전략이 필요하다.

    26.3.3 로그와 재현성 관리¶

    LLM 결과는 sampling 설정에 따라 달라질 수 있다. 실험 로그에는 model name, prompt, temperature, top-p, max tokens, retrieval 결과, tool call 결과를 남긴다. 문제가 생겼을 때 재현할 수 있어야 디버깅이 가능하다.

    정리¶

    개인 개발용 로컬 추론은 편의성과 학습이 우선이다. 작은 모델과 quantization으로 시작하고, RAG와 agent 실습에서는 prompt token 수와 로그를 꾸준히 확인한다. 로컬에서 얻은 감각은 나중에 팀 서버나 production serving을 이해하는 기반이 된다.
