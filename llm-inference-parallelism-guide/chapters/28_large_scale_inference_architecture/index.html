    28장. 대규모 서비스용 추론 아키텍처 맛보기¶

    대규모 LLM serving은 단일 서버를 크게 만든다고 해결되지 않는다. 모델별 worker pool, 요청 유형별 queue, multi-GPU 병렬화, multi-node deployment, SLO 기반 capacity planning이 필요하다. 이 장은 전체 그림을 맛보기로 정리한다.

    학습 목표¶

    • Multi-GPU serving에서 TP와 DP를 조합하는 이유를 이해한다.
    • Multi-node serving에서 네트워크가 중요한 이유를 설명한다.
    • Production checklist의 핵심 항목을 정리한다.

    28.1 Multi-GPU serving¶

    Multi-GPU serving에서는 여러 병렬화 축을 조합한다. 큰 모델 하나를 올리기 위해 TP를 쓰고, throughput을 늘리기 위해 DP replica를 늘릴 수 있다. Long-context 요청은 별도 worker pool로 분리할 수 있다.

    28.1.1 TP와 DP 조합¶

    예를 들어 8 GPU 서버에서 70B 모델 하나를 TP=4로 띄우고, 이런 replica를 2개 둘 수 있다. 이 경우 TP는 모델 하나를 나누는 역할을 하고, DP는 요청을 두 replica로 분산하는 역할을 한다.

    이처럼 TP와 DP는 서로 대체가 아니라 조합되는 경우가 많다.

    28.1.2 긴 context 요청 분리¶

    Long-context 요청은 KV cache를 많이 사용하고 TTFT가 길다. 짧은 chat 요청과 같은 worker pool에서 처리하면 tail latency가 악화될 수 있다. 긴 요청 전용 pool이나 별도 queue를 두면 SLO 관리가 쉬워진다.

    28.1.3 모델별 worker pool¶

    서비스가 여러 모델을 제공한다면 모델별 worker pool을 둔다. 작은 fast model, 큰 high-quality model, embedding model, reranker model은 서로 다른 latency와 resource 특성을 가진다.

    모델별 autoscaling과 routing policy가 필요해진다.

    28.2 Multi-node serving¶

    한 노드의 GPU로 부족하면 여러 노드를 묶어야 한다. 이때 네트워크가 성능의 핵심이 된다.

    28.2.1 빠른 네트워크가 중요하다¶

    TP나 DCP처럼 자주 통신하는 병렬화는 느린 네트워크에서 효율이 떨어진다. Multi-node에서는 어떤 통신이 노드 안에 머물고, 어떤 통신이 노드 사이를 지나가는지 설계해야 한다.

    28.2.2 TP는 통신 비용에 민감하다¶

    TP는 layer마다 collective communication이 발생할 수 있다. 가능하면 빠른 interconnect가 있는 같은 노드 안에서 TP를 구성하는 것이 유리하다.

    노드 간에는 PP나 DP가 더 적합한 경우가 많지만, 모델과 시스템에 따라 달라진다.

    28.2.3 PP와 DP가 노드 간 확장에 쓰일 수 있다¶

    Pipeline Parallel은 stage 간 activation을 전달하므로, 일부 경우 노드 간 확장에 사용할 수 있다. Data Parallel은 replica 간 통신이 적어 multi-node 확장에 비교적 단순하다.

    대규모 serving에서는 node topology와 병렬화 전략을 함께 설계한다.

    28.3 Production checklist¶

    Production LLM serving에서는 모델이 응답하는 것만으로 충분하지 않다. SLO, capacity, 배포 안정성, 비용 모니터링이 필요하다.

    28.3.1 SLO 정의¶

    TTFT, TPOT, end-to-end latency, availability, error rate 같은 SLO를 정의한다. SLO가 있어야 capacity planning과 tuning의 기준이 생긴다.

    28.3.2 Capacity planning¶

    예상 traffic, prompt length 분포, output length 분포, peak time, model별 사용량을 기준으로 필요한 GPU 수를 추정한다. 평균만 보면 안 되고 p95, p99를 함께 본다.

    28.3.3 Canary와 rollback¶

    모델 버전, quantization, serving engine 설정 변경은 품질과 latency에 영향을 준다. Canary 배포로 일부 traffic에 먼저 적용하고, 문제가 있으면 빠르게 rollback할 수 있어야 한다.

    28.3.4 비용 모니터링¶

    GPU hour, tokens served, cache hit rate, model별 cost, 사용자별 사용량을 모니터링한다. LLM serving 비용은 context length와 output length에 민감하므로 token 기반 비용 분석이 중요하다.

    정리¶

    대규모 LLM serving은 병렬화 전략과 운영 전략이 함께 필요하다. TP/DP/PP 조합, long-context 분리, 모델별 worker pool, SLO 기반 capacity planning, canary와 rollback, 비용 모니터링이 핵심이다.
