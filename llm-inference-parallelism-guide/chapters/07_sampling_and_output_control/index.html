    7장. Sampling과 출력 제어¶

    LLM은 다음 token을 하나만 "정답"으로 출력하지 않는다. 모델의 마지막 출력은 vocabulary에 있는 각 token 후보의 점수, 즉 logits다. Decoding parameter는 이 점수 분포에서 실제 token을 어떻게 고를지 결정한다.

    출력 제어는 품질 문제로만 보이지만, serving 비용과도 연결된다. 더 긴 출력을 유도하는 설정은 decode 반복을 늘리고, structured output이나 constrained decoding은 엔진에 추가 제약을 준다.

    학습 목표¶

    • Logits와 다음 token 선택 과정을 이해한다.
    • Temperature, top-k, top-p의 역할을 구분한다.
    • Structured output이 추론 비용에 미치는 영향을 이해한다.

    7.1 Logits와 token 선택¶

    모델은 매 decode step마다 다음 token 후보 전체에 대한 점수를 낸다. 이 점수가 logits다. Logits는 softmax를 거쳐 확률분포처럼 해석될 수 있고, decoding 전략은 이 분포에서 token을 선택한다.

    7.1.1 모델은 다음 token의 확률분포를 출력한다¶

    Vocabulary 크기가 100,000이라면 모델은 다음 token 후보 100,000개에 대한 점수를 낸다. 그중 가장 높은 점수를 가진 token을 고를 수도 있고, 확률적으로 sampling할 수도 있다. 같은 prompt라도 sampling 설정에 따라 답변이 달라질 수 있는 이유가 여기에 있다.

    7.1.2 Greedy decoding¶

    Greedy decoding은 매 단계에서 가장 점수가 높은 token을 선택한다. 결정적이고 재현성이 높다는 장점이 있다. 하지만 항상 가장 높은 확률의 token만 고르기 때문에 답변이 단조롭거나 반복적일 수 있다.

    테스트, 평가, 구조화된 출력처럼 재현성이 중요한 상황에서는 greedy에 가까운 설정이 유리할 수 있다.

    7.1.3 Sampling decoding¶

    Sampling decoding은 확률분포에서 token을 뽑는다. 확률이 높은 token이 더 자주 선택되지만, 항상 같은 token이 선택되지는 않는다. 창의적인 글쓰기, 다양한 후보 생성, brainstorming 같은 작업에서는 sampling이 더 자연스러울 수 있다.

    다만 sampling은 재현성을 낮출 수 있다. 운영 시스템에서는 temperature, top-p, seed, max output token을 명시적으로 기록해 문제 재현이 가능하도록 하는 것이 좋다.

    7.2 자주 쓰는 decoding parameter¶

    Decoding parameter는 모델 자체를 바꾸지 않는다. 같은 model weights와 같은 prompt라도 parameter 설정에 따라 token 선택 방식이 달라진다.

    7.2.1 Temperature¶

    Temperature는 확률분포를 얼마나 날카롭게 또는 평평하게 만들지 조절한다. 낮은 temperature는 높은 확률 token에 더 집중하게 만들고, 높은 temperature는 다양한 token이 선택될 가능성을 높인다.

    일반적으로 사실 기반 질의응답, 코드 생성, JSON 출력에서는 낮은 temperature가 선호된다. 창의적 글쓰기나 아이디어 생성에서는 조금 높은 temperature를 사용할 수 있다.

    7.2.2 Top-k와 top-p¶

    Top-k는 점수가 높은 k개의 token 후보만 남긴다. Top-p는 누적 확률이 p에 도달할 때까지의 token 후보만 남긴다. 둘 다 너무 낮은 확률의 token이 선택되는 것을 막기 위한 장치다.

    Top-k는 후보 개수를 고정하고, top-p는 분포 모양에 따라 후보 개수가 달라진다. 실무에서는 top-p를 더 자주 쓰는 경우가 많지만, 모델과 workload에 따라 적절한 값은 달라질 수 있다.

    7.2.3 Repetition penalty¶

    Repetition penalty는 같은 token이나 표현이 반복되는 것을 줄이기 위한 설정이다. LLM이 같은 문장을 반복하거나 목록을 끝없이 이어가는 경우에 도움이 될 수 있다.

    하지만 penalty가 너무 강하면 자연스럽게 반복되어야 하는 용어까지 억제할 수 있다. 예를 들어 기술 문서에서는 같은 핵심 용어가 반복되는 것이 정상일 수 있다. 따라서 품질 평가 없이 값을 크게 조정하는 것은 위험하다.

    7.3 Structured output과 추론 비용¶

    개발자는 LLM을 단순한 채팅이 아니라 JSON 생성, tool calling, agent workflow에 자주 사용한다. 이때 출력이 특정 schema를 따라야 한다면 decoding 과정에 제약이 들어간다.

    7.3.1 JSON 출력 강제¶

    JSON 출력은 애플리케이션 연동에 편리하다. 하지만 모델은 기본적으로 자유로운 token sequence를 생성한다. JSON을 강제하려면 prompt로 지시하거나, response format 기능을 사용하거나, constrained decoding을 적용한다.

    Prompt만으로 JSON을 요구하면 실패 가능성이 남는다. 따옴표가 빠지거나, trailing comma가 생기거나, schema와 다른 field가 나올 수 있다. 운영 시스템에서는 가능한 한 구조화된 출력 기능을 사용하는 것이 안전하다.

    7.3.2 Constrained decoding¶

    Constrained decoding은 다음 token 후보를 schema나 grammar에 맞는 token으로 제한하는 방식이다. 예를 들어 JSON object 안에서 key가 와야 하는 위치라면 가능한 token 후보를 제한할 수 있다.

    이 방식은 출력 안정성을 높이지만, sampling 과정에 추가 제약을 적용해야 하므로 구현 복잡도와 비용이 늘 수 있다. 특히 복잡한 grammar나 큰 schema에서는 decode 단계의 overhead를 고려해야 한다.

    7.3.3 Agent와 tool calling에서 structured generation이 중요한 이유¶

    Agent와 tool calling에서는 모델 출력이 다음 프로그램 동작을 결정한다. 함수 이름, 인자, JSON schema가 틀리면 단순한 문장 오류가 아니라 시스템 실행 오류가 된다. 따라서 structured generation은 품질 문제가 아니라 안정성 문제다.

    또한 agent workflow는 system prompt, tool schema, intermediate state가 반복되는 경우가 많다. 이때 prefix caching과 structured decoding이 함께 중요해진다.

    정리¶

    LLM은 매 단계에서 다음 token 후보들의 logits를 출력하고, decoding 전략이 실제 token을 선택한다. Temperature, top-k, top-p, repetition penalty는 답변의 다양성과 안정성을 조절한다.

    Structured output은 개발자에게 매우 중요하다. JSON과 tool calling은 LLM을 애플리케이션 로직에 연결하지만, decoding에 추가 제약을 만들고 비용에도 영향을 줄 수 있다.
