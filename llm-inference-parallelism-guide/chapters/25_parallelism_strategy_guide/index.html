    25장. 병렬화 전략 선택 가이드¶

    병렬화 전략은 GPU 수만 보고 정하지 않는다. 먼저 무엇이 병목인지 확인해야 한다. 모델이 GPU에 들어가지 않는 문제인지, 요청량이 많아 throughput이 부족한 문제인지, long-context 때문에 KV cache가 터지는 문제인지에 따라 선택지가 다르다.

    학습 목표¶

    • 모델이 GPU 하나에 들어가지 않을 때의 선택지를 정리한다.
    • 요청량이 많을 때 throughput 확장 방식을 판단한다.
    • Long-context 요청이 많을 때 필요한 전략을 이해한다.

    25.1 모델이 GPU 하나에 들어가지 않는 경우¶

    이 상황의 핵심은 weight memory다. 먼저 모델 weight와 실행에 필요한 추가 메모리가 단일 GPU에 들어가는지 확인한다.

    25.1.1 먼저 quantization을 검토한다¶

    가장 단순한 해결책은 더 낮은 precision을 사용하는 것이다. 검증된 INT8, INT4, FP8 quantized model을 사용하면 weight memory를 크게 줄일 수 있다. 단, 품질 저하 가능성이 있으므로 반드시 평가해야 한다.

    25.1.2 Tensor parallel을 검토한다¶

    Quantization만으로 부족하거나 FP16/BF16 품질이 필요하다면 Tensor Parallel을 검토한다. TP는 layer 내부 weight와 계산을 여러 GPU에 나눈다. 단일 GPU weight 한계를 넘는 데 효과적이다.

    빠른 interconnect가 있는 같은 노드 안에서 TP를 구성하는 것이 일반적으로 유리하다.

    25.1.3 Pipeline parallel을 보조적으로 검토한다¶

    모델이 매우 깊거나 여러 노드에 걸쳐 확장해야 한다면 Pipeline Parallel을 함께 고려할 수 있다. PP는 layer를 나누므로 TP와 다른 축으로 memory를 분산한다.

    단, pipeline bubble과 scheduling 복잡도가 있으므로 단순한 첫 선택지는 아니다.

    25.2 요청량이 많은 경우¶

    모델은 GPU에 잘 올라가지만 요청량이 많아 처리량이 부족한 경우에는 throughput 확장이 목표다.

    25.2.1 Data parallel을 검토한다¶

    모델이 GPU 한 장에 들어간다면 Data Parallel replica를 늘리는 것이 가장 단순하다. 여러 GPU 또는 여러 서버에 같은 모델을 띄우고 load balancer로 요청을 분산한다.

    25.2.2 Continuous batching을 활용한다¶

    Replica를 늘리기 전에 serving 엔진의 batching 설정이 적절한지 확인한다. Continuous batching은 GPU utilization과 throughput에 큰 영향을 준다.

    Batch를 키울 때는 p95 latency와 TPOT가 악화되지 않는지 함께 봐야 한다.

    25.2.3 Load balancing을 설계한다¶

    요청 길이가 다양하면 단순 round-robin이 비효율적일 수 있다. 긴 요청이 특정 replica에 몰리면 tail latency가 나빠진다. 가능하면 queue length, token budget, request class를 고려한 load balancing을 설계한다.

    25.3 Long-context 요청이 많은 경우¶

    Long-context 요청은 prefill과 KV cache 모두를 압박한다. 단순히 GPU를 늘리는 것보다 context 정책과 cache 전략을 먼저 봐야 한다.

    25.3.1 Max context length를 현실적으로 제한한다¶

    모델이 128K를 지원하더라도 모든 요청에 128K를 허용할 필요는 없다. 실제 사용량과 품질 개선 효과를 보고 tier별 context limit을 둘 수 있다.

    25.3.2 Prefix caching을 활용한다¶

    반복되는 system prompt, tool schema, RAG template이 있다면 prefix caching을 적용한다. Long-context에서 반복 prefix가 길수록 prefill 절감 효과가 커질 수 있다.

    25.3.3 Context parallel과 DCP를 검토한다¶

    KV cache가 batch size를 제한하고, context가 매우 긴 workload라면 Context Parallel 또는 DCP를 검토한다. Memory 이득과 통신 비용을 함께 측정해야 한다.

    정리¶

    전략 선택은 병목 진단에서 시작한다. 모델이 안 들어가면 quantization과 TP를 먼저 보고, 요청량이 많으면 DP와 batching을 본다. Long-context가 문제라면 context 제한, prefix caching, CP/DCP를 검토한다. 병렬화는 정답 하나가 아니라 workload별 trade-off다.

    점검 질문¶

    • 모델이 GPU에 안 들어가는 문제와 요청량이 많은 문제는 왜 다른 전략을 요구하는가?
    • Long-context 요청을 무조건 허용하면 어떤 비용이 발생하는가?
