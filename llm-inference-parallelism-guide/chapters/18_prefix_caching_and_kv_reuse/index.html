    18장. Prefix Caching과 KV Cache Reuse¶

    많은 LLM 서비스에서는 같은 prefix가 반복된다. System prompt, tool schema, RAG template, agent instruction이 대표적이다. 같은 token prefix를 매번 prefill하면 같은 계산을 반복하게 된다. Prefix caching은 이 반복 계산을 줄이기 위한 기법이다.

    학습 목표¶

    • Prefix가 반복되는 실제 workload를 식별한다.
    • Prefix caching이 TTFT와 비용에 미치는 효과를 이해한다.
    • Radix tree 기반 재사용의 직관을 설명한다.
    • KV cache reuse가 단일 서버를 넘어 여러 계층과 여러 worker로 확장될 수 있음을 이해한다.

    18.1 Prefix가 반복되는 상황¶

    Prefix caching은 "앞부분 token sequence가 동일한 요청"에서 효과가 있다. 텍스트가 비슷한 정도가 아니라 tokenizer 결과가 같은 prefix여야 한다는 점이 중요하다.

    18.1.1 System prompt¶

    챗봇이나 agent는 보통 고정 system prompt를 가진다. "너는 회사 내부 문서 도우미다" 같은 지시문이 모든 요청 앞에 붙는다. 이 부분은 여러 요청에서 반복되므로 prefix caching 후보가 된다.

    18.1.2 Multi-turn chat¶

    Multi-turn chat에서는 이전 대화 history가 다음 요청의 prefix가 될 수 있다. 같은 대화 세션 안에서 앞부분 history가 반복된다면, 일부 KV cache를 재사용할 수 있다.

    다만 대화가 매 turn마다 길어지므로 cache 관리와 eviction 정책이 중요해진다.

    18.1.3 RAG template¶

    RAG 시스템은 비슷한 prompt template을 반복 사용한다. 예를 들어 system instruction, 답변 형식, citation 규칙은 고정이고 검색된 문서와 질문만 바뀔 수 있다. 고정 prefix가 길수록 caching 효과가 커질 수 있다.

    18.1.4 Agent workflow¶

    Agent workflow에서는 tool schema, planner instruction, output format이 반복된다. 특히 tool list가 길면 매 요청마다 같은 schema를 prefill하는 비용이 커진다. Prefix caching은 이런 구조화된 LLM 프로그램에서 효과적이다.

    18.2 Prefix caching의 효과¶

    Prefix caching은 이미 계산한 prefix의 KV cache를 재사용해 prefill 비용을 줄인다. 결과적으로 TTFT와 GPU 비용을 낮출 수 있다.

    18.2.1 반복 prefill 계산을 줄인다¶

    같은 prefix를 매번 처음부터 모델에 통과시키는 것은 낭비다. Prefix caching은 prefix에 해당하는 KV cache를 저장해 두고, 다음 요청이 같은 prefix를 가지면 그 부분을 재사용한다.

    이렇게 하면 새 요청은 prefix 이후의 다른 부분만 추가로 prefill하면 된다.

    18.2.2 TTFT를 줄일 수 있다¶

    Prefill 시간이 줄면 첫 token까지의 시간도 줄 수 있다. 특히 긴 system prompt나 tool schema가 반복되는 서비스에서는 사용자가 체감하는 TTFT 개선이 클 수 있다.

    물론 cache lookup, memory 관리, cache miss 상황도 고려해야 한다. Cache hit rate가 낮으면 효과가 제한적이다.

    18.2.3 비용을 줄일 수 있다¶

    반복 계산이 줄어들면 GPU 연산량과 memory bandwidth 사용량도 줄 수 있다. 동일한 GPU로 더 많은 요청을 처리할 수 있으므로 비용 효율이 좋아질 수 있다.

    운영에서는 prefix cache hit rate, cache memory usage, eviction count 같은 지표를 관찰하는 것이 좋다.

    18.3 Radix tree 기반 재사용 직관¶

    Prefix caching을 효율적으로 하려면 공통 prefix를 잘 찾고 관리해야 한다. Radix tree는 문자열 또는 token sequence의 공통 prefix를 압축해 저장하는 자료구조다.

    18.3.1 공통 prefix를 tree로 관리한다¶

    여러 요청의 token sequence를 tree로 관리하면 공통으로 시작하는 부분을 공유할 수 있다. 예를 들어 system prompt가 같고 뒤쪽 질문만 다르면, system prompt에 해당하는 경로를 여러 요청이 공유한다.

    이 구조는 prefix가 길고 다양한 suffix가 붙는 agent/RAG workload에서 유리하다.

    18.3.2 동일한 token sequence의 KV cache를 공유한다¶

    중요한 것은 동일한 token sequence다. 공백 하나가 다르거나 template 버전이 바뀌면 token sequence가 달라질 수 있다. 그러면 cache hit가 깨질 수 있다.

    따라서 prompt template을 안정적으로 유지하고, 불필요한 동적 값을 prefix 앞부분에 넣지 않는 것이 좋다.

    18.3.3 구조화된 LLM 프로그램에서 효과가 크다¶

    SGLang 같은 시스템은 structured generation과 prefix reuse를 중요한 문제로 본다. Agent, RAG, tool calling은 반복되는 prefix와 달라지는 suffix가 뚜렷하므로 cache reuse의 여지가 크다.

    LLM 애플리케이션을 설계할 때도 cache-friendly prompt 구조를 고려할 수 있다. 고정 instruction은 앞에 두고, 자주 바뀌는 값은 뒤로 보내면 prefix reuse 가능성이 커진다.

    18.4 계층형 KV cache와 reuse 시스템¶

    Prefix caching을 단일 프로세스 안의 최적화로만 볼 필요는 없다. Long-context와 agent workflow에서는 같은 문서, 같은 system prompt, 같은 tool schema가 여러 요청과 여러 worker에서 반복될 수 있다. 이때 KV cache를 GPU HBM 안에만 두면 용량이 부족하고, 요청이 다른 worker로 가면 이전 cache를 활용하기 어렵다.

    최근 시스템은 KV cache를 별도의 reuse layer로 다루는 방향으로 발전하고 있다. 예를 들어 LMCache 같은 계층은 inference engine이 만든 KV cache를 GPU 밖의 CPU RAM, local storage, remote backend에 저장하고, 이후 같은 token 구간이 필요할 때 다시 가져와 prefill 계산을 줄이는 방식을 제공한다.

    18.4.1 Reuse의 단위는 텍스트가 아니라 token 구간이다¶

    사용자 입장에서는 같은 문서를 다시 넣었다고 생각해도, 실제 cache reuse는 tokenizer 결과와 chunking 방식에 의해 결정된다. 같은 문장이라도 template, 공백, system prompt 버전, 문서 삽입 위치가 바뀌면 token sequence가 달라질 수 있다.

    따라서 계층형 KV cache를 잘 활용하려면 prompt template을 안정적으로 유지하고, 반복되는 큰 prefix가 앞쪽에 오도록 설계하는 것이 중요하다.

    18.4.2 저장 계층이 커질수록 hit와 이동 비용을 함께 봐야 한다¶

    GPU HBM의 cache hit는 빠르지만 용량이 작다. CPU RAM이나 local SSD는 더 많은 cache를 보관할 수 있지만, 다시 GPU로 가져오는 비용이 든다. Remote storage는 더 큰 공유 cache를 만들 수 있지만 network latency와 bandwidth가 중요해진다.

    그래서 운영에서는 cache hit rate만 보면 부족하다. Hit가 나더라도 느린 계층에서 가져오는 시간이 길면 사용자 latency가 나빠질 수 있다. 반대로 긴 prompt를 다시 prefill하는 비용이 훨씬 크다면, 느린 계층에서 cache를 가져오는 편이 이득일 수 있다.

    18.4.3 관찰해야 할 지표가 늘어난다¶

    Prefix caching을 운영 기능으로 다루려면 다음 지표를 함께 봐야 한다.

    • prefix cache hit rate
    • 계층별 cache 사용량
    • HBM에서 CPU RAM 또는 SSD로 내려간 cache 양
    • cache load/offload latency
    • cache eviction count
    • cache hit가 TTFT와 throughput에 미치는 영향

    이 지표들은 "cache가 켜져 있다"보다 훨씬 중요하다. 실제 workload에서 반복되는 prefix가 충분히 있고, cache 이동 비용보다 재계산 비용이 클 때 계층형 reuse가 의미를 가진다.

    정리¶

    Prefix caching은 반복되는 token prefix의 KV cache를 재사용해 prefill 비용을 줄이는 기법이다. System prompt, chat history, RAG template, agent tool schema에서 효과가 크다. Cache hit를 높이려면 token sequence가 안정적으로 반복되도록 prompt 구조를 설계해야 한다.

    규모가 커지면 KV cache reuse는 GPU HBM 안의 작은 최적화를 넘어 CPU RAM, local SSD, remote storage까지 포함하는 계층형 cache 문제로 확장된다. 이때는 hit rate뿐 아니라 cache 이동 비용과 eviction 정책까지 함께 봐야 한다.

    점검 질문¶

    • Prefix caching은 어떤 종류의 prompt에서 효과가 큰가?
    • Prefix가 텍스트로 같아 보여도 token sequence가 다르면 어떤 문제가 생기는가?
    • 계층형 KV cache에서 hit rate만으로 성능을 판단하기 어려운 이유는 무엇인가?
