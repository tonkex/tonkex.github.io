    13장. Long-context 추론의 어려움¶

    Long-context 모델은 긴 문서, 긴 대화, 대규모 코드 컨텍스트를 한 번에 넣을 수 있게 해 준다. 개발자 입장에서는 매우 편리하다. 하지만 context window가 커졌다고 해서 비용이 사라지는 것은 아니다. 긴 prompt는 prefill을 느리게 만들고, 긴 KV cache는 decode와 batch size를 압박한다.

    Long-context 추론을 이해하려면 "넣을 수 있는 길이"와 "효율적으로 처리할 수 있는 길이"를 구분해야 한다.

    학습 목표¶

    • 긴 prompt가 TTFT와 attention 계산에 미치는 영향을 이해한다.
    • 긴 KV cache가 decode와 batch size에 주는 부담을 설명한다.
    • 개발자가 자주 겪는 long-context 문제를 시스템 관점에서 해석한다.

    13.1 긴 prompt의 비용¶

    긴 prompt는 먼저 prefill 비용을 증가시킨다. Prefill은 prompt 전체를 읽고 KV cache를 만드는 단계이므로, prompt token 수가 많을수록 첫 token까지의 시간이 늘어난다.

    13.1.1 TTFT가 증가한다¶

    사용자가 긴 문서를 넣고 "요약해 줘"라고 요청하면, 모델은 답변을 시작하기 전에 문서 전체를 처리해야 한다. 이 시간이 TTFT에 반영된다. Streaming을 켜도 첫 token이 나오기 전까지는 사용자가 기다려야 한다.

    따라서 long-context 서비스에서는 TTFT를 별도로 측정해야 한다. End-to-end latency만 보면 prefill 병목인지 decode 병목인지 알기 어렵다.

    13.1.2 Attention 계산이 커진다¶

    Prefill에서는 prompt 안의 token들이 서로를 참조한다. Sequence length가 길어질수록 attention 계산과 메모리 접근이 커진다. FlashAttention 같은 최적화가 중요해지는 이유가 여기에 있다.

    긴 prompt는 단순히 token 수만 늘리는 것이 아니라, attention 작업을 더 어렵게 만든다. 특히 아주 긴 context에서는 attention kernel, memory layout, chunked prefill 같은 엔진 최적화가 성능 차이를 만든다.

    13.1.3 Tokenization 비용도 무시할 수 없다¶

    대규모 serving에서는 tokenization도 병목이 될 수 있다. 긴 문서나 많은 동시 요청을 CPU에서 tokenization해야 하므로, GPU 계산 전 단계에서 queue가 생길 수 있다.

    한국어, 코드, JSON, 로그처럼 token 수가 많이 나오는 입력은 tokenization 비용과 prefill 비용을 모두 키울 수 있다. 운영에서는 tokenizer CPU 사용률과 request queueing도 함께 봐야 한다.

    13.2 긴 KV cache의 비용¶

    Long-context의 두 번째 비용은 KV cache다. Prefill이 끝난 뒤에도 긴 context는 사라지지 않는다. Decode 단계에서 새 token을 만들 때마다 과거 context의 KV cache를 참조해야 한다.

    13.2.1 Decode 때도 긴 context를 계속 참조한다¶

    새 output token의 Query는 과거 token들의 Key와 Value를 참조한다. Context가 4K이면 4K에 가까운 과거 상태를 보고, context가 128K이면 훨씬 긴 cache를 본다. 따라서 long-context는 prefill뿐 아니라 decode에도 영향을 준다.

    짧은 답변이라도 입력 context가 매우 길면 decode 단계의 cache 접근 비용이 커질 수 있다.

    13.2.2 GPU memory를 많이 차지한다¶

    KV cache는 layer마다 저장되고, sequence length에 비례해 커진다. Long-context 요청 하나가 많은 GPU memory를 점유할 수 있다. 여기에 batch size가 곱해지면 동시 처리 가능한 요청 수가 급격히 줄어든다.

    이 때문에 long-context 모델을 serving할 때는 max model length를 신중하게 설정해야 한다. 모델이 128K를 지원하더라도 모든 사용자에게 항상 128K를 열어 두는 것은 비용이 크다.

    13.2.3 Batch size를 제한한다¶

    GPU memory가 KV cache로 채워지면 더 많은 요청을 batch에 넣을 수 없다. Batch size가 작아지면 throughput이 떨어질 수 있다. 긴 요청 하나가 worker의 memory budget을 크게 차지하면 짧은 요청들의 latency까지 악화될 수 있다.

    실무에서는 긴 context 요청을 별도 queue나 worker pool로 분리하는 전략도 사용한다. 모든 요청을 같은 path로 처리하면 tail latency 관리가 어려워진다.

    13.3 개발자가 자주 겪는 long-context 문제¶

    Long-context 문제는 RAG, multi-turn chat, 코드 분석, agent workflow에서 자주 나타난다. 겉으로는 "모델이 느리다"로 보이지만, 실제로는 prompt 설계와 serving 설정 문제가 섞여 있다.

    13.3.1 RAG 문서를 많이 넣었더니 첫 응답이 느리다¶

    검색된 문서를 많이 넣으면 prompt token 수가 늘어난다. 이때 prefill 시간이 길어져 TTFT가 증가한다. 문서가 관련성이 낮거나 중복된다면 비용만 늘고 답변 품질 개선은 제한적일 수 있다.

    대응 방법은 retrieval 품질 개선, chunk 중복 제거, reranking, prompt compression, prefix caching, chunked prefill 검토 등이 있다.

    13.3.2 긴 대화가 이어질수록 비용이 증가한다¶

    Multi-turn chat에서는 이전 대화 history가 계속 prompt에 포함될 수 있다. 대화가 길어질수록 input token 수가 늘고, KV cache도 커진다. "처음에는 빨랐는데 대화가 길어질수록 느려진다"는 현상은 자연스러운 결과다.

    대응 방법은 대화 요약, 오래된 메시지 제거, 중요 정보만 memory로 유지, max history token 제한 등이 있다.

    13.3.3 동시 요청이 늘면 갑자기 OOM이 발생한다¶

    단일 long-context 요청은 처리되지만, 여러 개가 동시에 들어오면 OOM이 날 수 있다. 각 요청의 KV cache가 합쳐져 GPU memory를 초과하기 때문이다.

    이때는 max concurrent requests보다 max batched tokens, max model length, GPU memory utilization을 함께 봐야 한다. 요청 수만 제한해도 각 요청이 너무 길면 충분하지 않을 수 있다.

    정리¶

    Long-context는 기능적으로 강력하지만 prefill latency, KV cache memory, batch size 제약을 함께 만든다. 긴 prompt는 TTFT를 늘리고, 긴 KV cache는 decode와 동시성에 부담을 준다. 개발자는 context length를 품질 개선 수단으로만 보지 말고, GPU memory와 latency 예산을 쓰는 선택으로 봐야 한다.
