    20장. vLLM 입문¶

    vLLM은 LLM serving을 입문자가 가장 쉽게 실습해 볼 수 있는 도구 중 하나다. OpenAI-compatible API server를 제공하고, PagedAttention 기반의 KV cache 관리와 continuous batching을 통해 여러 요청을 효율적으로 처리한다.

    이 장의 목표는 vLLM의 모든 옵션을 외우는 것이 아니다. vLLM을 통해 앞에서 배운 prefill, decode, KV cache, batching, tensor parallelism이 실제 serving 명령어와 어떻게 연결되는지 감을 잡는 것이다.

    학습 목표¶

    • vLLM이 해결하는 serving 문제를 이해한다.
    • 기본 API server 실행 흐름을 파악한다.
    • 병렬화 설정의 의미를 맛본다.

    20.1 vLLM이 해결하는 문제¶

    Hugging Face Transformers의 generate()를 직접 호출해도 LLM 추론은 가능하다. 하지만 다수 사용자에게 API 형태로 serving하려면 요청 scheduling, KV cache 관리, batching, streaming, OpenAI 호환 API 같은 기능이 필요하다. vLLM은 이런 serving 문제를 해결하기 위해 만들어진 엔진이다.

    20.1.1 빠른 LLM serving¶

    vLLM은 단일 요청 실행뿐 아니라 여러 요청을 동시에 처리하는 serving workload를 겨냥한다. Continuous batching으로 GPU를 효율적으로 사용하고, attention/KV cache 최적화를 통해 throughput을 높인다.

    입문자는 vLLM을 사용하면서 "모델을 호출한다"에서 "모델 서버를 운영한다"로 관점을 확장할 수 있다.

    20.1.2 PagedAttention 기반 KV cache 관리¶

    vLLM의 대표적인 아이디어는 PagedAttention이다. KV cache를 block/page처럼 관리해 요청마다 길이가 다른 serving 환경에서 메모리 낭비를 줄인다. 이 개념은 6장에서 살펴본 것처럼 여러 요청의 KV cache를 효율적으로 관리하는 데 중요하다.

    즉 vLLM의 강점은 단순히 모델 연산이 빠른 것만이 아니라, 많은 요청이 들어오는 상황에서 GPU memory를 잘 관리한다는 데 있다.

    20.1.3 OpenAI-compatible API server¶

    vLLM은 OpenAI API와 비슷한 형태의 HTTP endpoint를 제공한다. 기존 애플리케이션이 OpenAI SDK나 API 형식에 맞춰져 있다면, self-hosted model로 전환할 때 client code 변경을 줄일 수 있다.

    예를 들어 chat completions API를 사용하던 서비스는 base URL만 바꿔 vLLM server를 호출하는 구조로 실습할 수 있다.

    20.2 기본 실행¶

    실제 명령어는 vLLM 버전에 따라 조금씩 달라질 수 있으므로 공식 문서를 확인해야 한다. 현재 vLLM 문서의 기본 흐름은 vllm serve <MODEL> 형태다.

    20.2.1 모델 다운로드¶

    vLLM은 Hugging Face model repository나 로컬 model path를 사용할 수 있다. Gated model은 Hugging Face token이 필요할 수 있다. 입문 실습에서는 너무 큰 모델보다 7B 이하 또는 더 작은 instruct model로 시작하는 것이 좋다.

    모델 선택 기준은 다음과 같다.

    • GPU memory에 model weights와 KV cache가 들어가는가?
    • tokenizer와 chat template이 잘 지원되는가?
    • 상업적 사용이나 배포 license가 목적에 맞는가?

    20.2.2 단일 GPU 서버 실행¶

    가장 단순한 실행 예시는 다음과 같다.

    vllm serve facebook/opt-13b

    실제 실습에서는 사용할 GPU memory와 모델 크기를 고려해 더 작은 모델을 선택할 수 있다. 서버가 뜨면 OpenAI-compatible endpoint로 요청을 보낼 수 있다.

    운영 설정에서는 --max-model-len, --gpu-memory-utilization, --dtype, --quantization 같은 옵션을 검토하게 된다. 처음에는 옵션을 최소화하고, 정상 응답과 token 수, latency를 확인한 뒤 하나씩 조정하는 것이 좋다.

    20.2.3 Python client로 호출¶

    OpenAI-compatible server를 띄웠다면 Python client에서 base URL을 self-hosted server로 바꿔 호출할 수 있다. 예시는 다음과 같은 형태다.

    from openai import OpenAI

client = OpenAI(
    base_url="http://localhost:8000/v1",
    api_key="EMPTY",
)

response = client.chat.completions.create(
    model="facebook/opt-13b",
    messages=[{"role": "user", "content": "KV cache를 한 문장으로 설명해 줘."}],
    max_tokens=128,
)

print(response.choices[0].message.content)

    입문 실습에서는 먼저 non-streaming으로 확인하고, 이후 streaming을 켜서 TTFT와 token 출력 간격을 관찰하면 좋다.

    20.3 병렬화 설정 맛보기¶

    vLLM은 multi-GPU serving에서 Tensor Parallel과 Pipeline Parallel 설정을 제공한다. 공식 문서 기준으로 multi-GPU serving에서는 --tensor-parallel-size를 사용한다. Pipeline Parallel을 함께 쓰려면 --pipeline-parallel-size를 추가한다.

    20.3.1 Tensor parallel size¶

    Tensor Parallel을 4 GPU로 사용하는 예시는 다음과 같은 형태다.

    vllm serve facebook/opt-13b \
  --tensor-parallel-size 4

    이 설정은 layer 내부 계산과 weight를 4개 GPU에 나누겠다는 의미다. 모델이 GPU 한 장에 들어가지 않거나, 큰 모델의 계산을 여러 GPU로 나누고 싶을 때 사용한다.

    20.3.2 Pipeline parallel size¶

    Tensor Parallel과 Pipeline Parallel을 조합하는 예시는 다음과 같다.

    vllm serve gpt2 \
  --tensor-parallel-size 4 \
  --pipeline-parallel-size 2

    이 경우 전체 GPU 수는 보통 tensor_parallel_size x pipeline_parallel_size로 생각한다. 예시는 개념 설명용이며, 실제 모델과 GPU 구성에 맞게 조정해야 한다.

    20.3.3 Data parallel serving 개념¶

    vLLM에서도 대규모 serving에서는 여러 replica를 두고 요청을 분산하는 Data Parallel 구성이 필요할 수 있다. 작은 모델이 GPU 하나에 잘 들어간다면 여러 vLLM server instance를 띄우고 load balancer를 앞에 두는 방식으로 시작할 수 있다.

    큰 모델에서는 TP/PP로 하나의 replica를 구성하고, 이런 replica group을 여러 개 두는 식으로 확장한다.

    정리¶

    vLLM은 입문자가 LLM serving 개념을 실습하기 좋은 도구다. PagedAttention, continuous batching, OpenAI-compatible API server를 통해 KV cache 관리와 serving scheduling의 중요성을 체감할 수 있다. Multi-GPU에서는 --tensor-parallel-size, 필요 시 --pipeline-parallel-size를 통해 병렬화 개념을 실험할 수 있다.
