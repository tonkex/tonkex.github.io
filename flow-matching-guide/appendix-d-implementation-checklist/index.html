    부록 D. 구현 체크리스트¶

    최초 작성: 2026-05-23 02:18 KST 최종 업데이트: 2026-05-23 08:54 KST

    부록의 목적¶

    이 부록은 Flow Matching을 직접 구현하거나 논문 코드를 읽을 때 확인해야 할 항목을 모은 체크리스트다. 구현 오류는 대부분 복잡한 수식보다 shape, normalization, 시간 처리, target sign, sampler 설정에서 생긴다.

    1. 데이터와 전처리¶

    • 데이터가 어떤 범위로 normalize되는지 확인한다.
    • 이미지라면 \([0,1]\)인지 \([-1,1]\)인지 명확히 기록한다.
    • 로봇 action이라면 joint position, velocity, end-effector pose, gripper command의 scale을 따로 확인한다.
    • 학습 데이터와 sampling 결과를 같은 기준으로 de-normalize하는지 확인한다.
    • train/validation split과 random seed를 기록한다.

    2. Path Sampler¶

    • \(x_0\)와 \(x_1\)의 shape가 같은지 확인한다.
    • 시간 \(t\)가 batch별로 샘플링되는지 확인한다.
    • \(t\)를 broadcast할 shape가 올바른지 확인한다.
    • \(t=0\)에서 \(x_t=x_0\)인지 확인한다.
    • \(t=1\)에서 \(x_t=x_1\)인지 확인한다.
    • 직선 경로라면 target velocity가 \(x_1-x_0\)인지 확인한다.
    • 경로가 바뀌면 \(x_t\)와 \(u_t\)를 함께 바꾸었는지 확인한다.

    3. 모델 입력과 출력¶

    • 모델 입력이 \(x_t\)와 \(t\)를 모두 포함하는지 확인한다.
    • 조건부 모델이라면 class, text, image, robot state 같은 조건 \(c\) 또는 \(o\)가 실제로 모델에 들어가는지 확인한다.
    • 모델 출력 shape가 target velocity shape와 같은지 확인한다.
    • 시간 임베딩이 모든 필요한 block에 전달되는지 확인한다.
    • 로봇 action chunk라면 출력 shape가 \(B\times H\times d_a\)인지 확인한다.

    4. Loss¶

    • MSE가 prediction과 target velocity 사이에 계산되는지 확인한다.
    • target velocity에 불필요한 gradient가 흐르지 않는지 확인한다.
    • reduction이 batch, channel, spatial 또는 action dimension에서 의도대로 적용되는지 확인한다.
    • 시간 구간별 loss를 따로 기록할 수 있으면 좋다.
    • loss 값만 보지 말고 샘플과 trajectory를 함께 본다.

    5. Sampling¶

    • sampling 시작점이 \(p_0\)에서 온 noise인지 확인한다.
    • ODE time grid가 \(0\)에서 \(1\)로 진행되는지 확인한다.
    • reverse 방향을 써야 하는 설정이라면 time direction을 명확히 기록한다.
    • Euler, midpoint, RK 계열 중 어떤 solver를 쓰는지 기록한다.
    • step 수와 NFE를 함께 기록한다.
    • 같은 seed에서 step 수만 바꿔 비교한다.

    6. 시각화와 디버깅¶

    • 2D toy example에서는 velocity field quiver plot을 그린다.
    • 이미지 예제에서는 fixed noise grid로 checkpoint별 샘플을 비교한다.
    • 로봇 예제에서는 action trajectory, end-effector path, acceleration profile을 본다.
    • target velocity norm과 predicted velocity norm을 시간별로 기록한다.
    • \(t\)별 loss가 특정 구간에서만 높은지 확인한다.

    7. 로보틱스 적용 체크리스트¶

    • action representation이 무엇인지 명확히 한다.
    • action chunk 길이 \(H\)와 action dimension \(d_a\)를 기록한다.
    • 관측 \(o\)에 어떤 센서 정보와 언어 지시가 들어가는지 기록한다.
    • inference latency가 제어 주기를 만족하는지 측정한다.
    • trajectory smoothness, acceleration, jerk를 확인한다.
    • collision, joint limit, workspace constraint를 확인한다.
    • real robot 검증 여부와 simulation-only 결과를 구분한다.
    • 실패 시 fallback controller나 safety stop이 있는지 확인한다.

    8. 재현성 기록¶

    • dataset version
    • model architecture
    • path type
    • optimizer와 learning rate
    • batch size
    • training steps
    • random seed
    • solver type
    • NFE 또는 sampling steps
    • checkpoint 선택 기준
    • 평가 metric과 시각화 seed

    9. 최소 테스트 예시¶

    구현 초기에 다음 테스트를 통과시키면 많은 오류를 줄일 수 있다.

    def test_linear_path_boundary(path, x0, x1):
    t0 = torch.zeros(x0.shape[0], device=x0.device)
    t1 = torch.ones(x0.shape[0], device=x0.device)
    xt0, ut0 = path.sample(x0, x1, t0)
    xt1, ut1 = path.sample(x0, x1, t1)
    assert torch.allclose(xt0, x0)
    assert torch.allclose(xt1, x1)
    assert torch.allclose(ut0, x1 - x0)
    assert torch.allclose(ut1, x1 - x0)

    이 테스트는 성능을 보장하지는 않지만, 경로 방향과 boundary condition 오류를 초기에 잡아 준다.
