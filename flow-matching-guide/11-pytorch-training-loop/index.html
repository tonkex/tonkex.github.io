    11장. PyTorch Training Loop 설계¶

    최초 작성: 2026-05-23 01:43 KST 최종 업데이트: 2026-05-23 08:54 KST

    이 장의 질문¶

    Toy example을 넘어 실제 실험 코드를 만들려면 무엇을 분리해야 할까? Flow Matching 학습 루프는 짧게 쓸 수 있지만, 책이나 논문을 따라 여러 경로와 모델을 실험하려면 구조가 금방 복잡해진다. 이 장은 재사용 가능한 PyTorch 코드 구조를 설계하는 데 초점을 둔다.

    목표는 거대한 프레임워크를 만드는 것이 아니다. 경로, 모델, 데이터, solver, logging을 적절히 분리해서 실험이 망가졌을 때 원인을 찾기 쉽게 만드는 것이다.

    최소 구성 요소¶

    Flow Matching 실험 코드는 다음 구성 요소로 나누면 좋다.

    • dataset 또는 dataloader
    • noise sampler
    • path sampler
    • velocity model
    • loss function
    • optimizer와 scheduler
    • sampler 또는 ODE solver
    • evaluation과 visualization
    • checkpoint와 config

    이 중 학습 루프의 핵심은 path sampler와 velocity model의 연결이다.

    \[ (x_0,x_1,t)\mapsto(x_t,u_t), \]
    \[ (x_t,t)\mapsto v_\theta(x_t,t). \]

    그리고 loss는

    \[ \left\|v_\theta(x_t,t)-u_t\right\|^2 \]

    이다.

    PathSampler 분리¶

    경로 수식은 실험의 핵심 변수다. 따라서 학습 루프 안에 직접 쓰기보다 클래스로 분리하는 편이 좋다.

    class LinearPath:
    def sample(self, x0, x1, t):
        view_shape = [x0.shape[0]] + [1] * (x0.ndim - 1)
        t_view = t.view(*view_shape)
        xt = (1 - t_view) * x0 + t_view * x1
        ut = x1 - x0
        return xt, ut

    이 구조의 장점은 나중에 다른 경로를 추가해도 trainer 코드를 거의 바꾸지 않아도 된다는 점이다.

    xt, ut = path.sample(x0, x1, t)
pred = model(xt, t)
loss = mse(pred, ut)

    13장에서 여러 path parameterization을 비교할 때 이 분리가 중요해진다.

    시간 임베딩¶

    모델이 시간 \(t\)를 이해하려면 시간 정보를 적절히 넣어야 한다. toy example에서는 단순 concatenate로 충분할 수 있지만, 이미지 모델에서는 sinusoidal embedding이나 learned embedding을 쓴다.

    시간 임베딩의 목적은 모델이 다음과 같은 차이를 구분하게 하는 것이다.

    • \(t\)가 작을 때: 노이즈에 가까운 상태
    • \(t\)가 중간일 때: 구조가 섞인 상태
    • \(t\)가 클 때: 데이터에 가까운 상태

    수식으로는 모델을 다음처럼 볼 수 있다.

    \[ v_\theta(x_t,t)=f_\theta(x_t,\phi(t)), \]

    여기서 \(\phi(t)\)는 시간 임베딩 함수다.

    시간 임베딩이 약하면 모델은 같은 \(x_t\)라도 시간에 따라 다른 속도가 필요하다는 사실을 표현하기 어렵다.

    Trainer의 한 step¶

    학습 step은 다음 순서를 유지한다.

    def train_step(batch):
    x1 = batch.to(device)
    x0 = torch.randn_like(x1)
    t = torch.rand(x1.shape[0], device=device)
    xt, target_v = path.sample(x0, x1, t)
    pred_v = model(xt, t)
    loss = ((pred_v - target_v) ** 2).mean()
    optimizer.zero_grad(set_to_none=True)
    loss.backward()
    optimizer.step()
    return loss.item()

    여기서 batch는 데이터 샘플 \(x_1\)이다. 시작 샘플 \(x_0\)는 매 step마다 새로 뽑는다. 이 방식은 같은 데이터라도 다양한 노이즈 출발점과 연결되게 한다.

    Config로 관리할 항목¶

    실험을 반복하려면 config가 필요하다. 최소한 다음 항목을 기록해야 한다.

    • dataset 이름과 preprocessing
    • batch size
    • learning rate
    • optimizer 종류
    • training steps 또는 epochs
    • path type
    • time sampling 방식
    • model size
    • random seed
    • ODE solver 종류와 step 수
    • checkpoint 저장 주기

    Flow Matching은 경로와 solver가 결과에 큰 영향을 주므로, 이 둘을 config에 반드시 남겨야 한다. loss만 저장하고 경로 설정을 잃어버리면 나중에 결과를 재현하기 어렵다.

    Logging과 시각화¶

    로그에는 loss만 남기면 부족하다. 다음 항목을 함께 확인하는 것이 좋다.

    • training loss moving average
    • validation 또는 held-out loss
    • sample image 또는 toy scatter plot
    • velocity norm의 평균과 분산
    • ODE step 수별 sample 품질
    • 시간 구간별 loss

    시간 구간별 loss는 특히 유용하다. 예를 들어 \(t\in[0,0.25]\), \([0.25,0.5]\), \([0.5,0.75]\), \([0.75,1]\)로 나누어 loss를 보면 모델이 어느 시간대에서 어려움을 겪는지 알 수 있다.

    Checkpoint 설계¶

    checkpoint에는 모델 가중치만 저장하지 말고 실험 상태를 함께 저장해야 한다.

    • model state
    • optimizer state
    • training step
    • config
    • random seed 또는 RNG state
    • exponential moving average를 쓴다면 EMA state

    샘플 품질을 볼 때는 보통 EMA 모델이 더 안정적일 수 있다. 입문 구현에서는 필수는 아니지만, 이미지 실험으로 넘어가면 고려할 만하다.

    테스트 가능한 작은 단위¶

    Flow Matching 코드는 다음 단위를 테스트할 수 있다.

    1. path sampler가 올바른 shape를 반환하는가?
    2. \(t=0\)일 때 \(x_t=x_0\)인가?
    3. \(t=1\)일 때 \(x_t=x_1\)인가?
    4. 직선 경로에서 \(u_t=x_1-x_0\)인가?
    5. 모델 출력 shape가 입력 shape와 같은가?
    6. ODE sampler가 step 수에 맞게 \(t\)를 증가시키는가?

    이런 테스트는 성능을 보장하지는 않지만, 구현 오류를 초기에 잡아 준다.

    그림 후보¶

    • 코드 모듈 구조도: dataset, path sampler, model, trainer, solver, logger가 어떻게 연결되는지 보여 준다.
    • training step 흐름도: batch에서 loss까지 이어지는 텐서 흐름을 단계별로 표시한다.
    • 시간 구간별 loss plot: 전체 loss 하나보다 시간 구간별 난이도를 보여 주는 그래프를 제안한다.

    요약¶

    • PyTorch 구현에서는 path sampler, model, trainer, solver를 분리하면 실험 확장이 쉬워진다.
    • 학습 step의 핵심은 \(x_0\), \(x_1\), \(t\)에서 \(x_t\)와 목표 속도 \(u_t\)를 만들고 모델 출력과 MSE를 계산하는 것이다.
    • 시간 임베딩은 모델이 시각별로 다른 속도장을 표현하도록 돕는다.
    • config에는 path type과 ODE solver 설정을 반드시 남겨야 재현이 가능하다.
    • shape, boundary condition, velocity sign, sampler time grid는 작은 테스트로 확인하는 것이 좋다.
