    AI 인프라 엔지니어 로드맵¶

    초안 공개 준비: 2026-07-08 KST 현재 단계: 구조 설계 완료, 샘플 챕터 작성 전

    이 책은 LLM API를 사용해 본 backend/platform developer가 AI infrastructure engineer로 확장될 때 필요한 시스템 지도를 제공하기 위해 준비 중인 웹북이다.

    핵심 질문은 단순하다. LLM serving은 왜 느리고 비싼가, GPU 메모리는 어디에 쓰이는가, KV cache는 왜 capacity를 지배하는가, 그리고 vLLM, TensorRT-LLM, SGLang, Kubernetes, observability, evaluation, MLOps는 하나의 운영 흐름 안에서 어떻게 연결되는가.

    현재 공개 범위

    이 페이지는 완성 원고가 아니라 책의 공개용 골격이다. 원자료와 개인 작업 메모는 공개하지 않는다. 본문 초안은 샘플 챕터 검증 후 docs/에 추가한다.

    대상 독자¶

    • LLM API 사용 경험이 있는 backend/platform developer
    • AI infrastructure 역할로 확장하려는 ML engineer 또는 software engineer
    • Docker, HTTP API, Linux는 익숙하지만 GPU, LLM serving, Kubernetes GPU 운영, observability는 조각나게 이해하고 있는 독자

    읽고 나면 답해야 하는 질문¶

    • 지금 느린 원인이 모델 크기, KV cache, scheduler, network, storage, application trace 중 어디에 있는가?
    • TTFT, TPOT, throughput, p95/p99, cost per token은 각각 무엇을 보여주는가?
    • vLLM, TensorRT-LLM, SGLang, llama.cpp, Triton, KServe, Ray Serve는 어떤 기준으로 선택해야 하는가?
    • 단일 GPU 실험 결과를 Kubernetes production deployment로 확장할 때 무엇을 새로 측정해야 하는가?
    • LLM application release에서 model weight 외에 무엇을 versioning하고 관측해야 하는가?

    구성 방향¶

    이 책은 도구 목록이 아니라 운영자가 부딪히는 병목의 순서로 구성한다.

    1. 토큰, context length, batch가 비용 단위가 되는 이유를 설명한다.
    2. GPU, VRAM, CUDA memory hierarchy, KV cache를 통해 serving capacity를 계산한다.
    3. 단일 GPU에서 vLLM 기준 serving을 측정한다.
    4. TensorRT-LLM, SGLang, llama.cpp를 workload fit으로 비교한다.
    5. multi-GPU, NCCL, Kubernetes, KServe로 platform 계층을 확장한다.
    6. RAG, agent workload, observability, evaluation, security, MLOps를 production release loop 안에 배치한다.

    현재 추천 샘플 챕터¶

    첫 샘플 챕터는 KV cache가 LLM serving의 중심 병목이 되는 순간 이다.

    이 장은 책의 핵심 스타일을 검증하기에 가장 좋다. 실무 문제에서 시작해 prefill/decode, KV cache 크기 공식, PagedAttention, prefix caching, KV quantization trade-off까지 연결할 수 있기 때문이다.
