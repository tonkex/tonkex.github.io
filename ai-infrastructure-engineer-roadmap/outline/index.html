    전체 목차¶

    이 목차는 2026-07-08 기준의 구조 설계안이다. 본문 작성과 검증 과정에서 장 제목과 순서는 바뀔 수 있다.

    1부. AI 인프라 엔지니어의 지도¶

    1장. AI 인프라 엔지니어는 무엇을 운영하는가¶

    LLM API 뒤에 있는 model, runtime, GPU, serving API, data, evaluation, security, observability를 하나의 운영 시스템으로 정의한다.

    2장. 토큰, 컨텍스트, 배치가 비용 단위가 되는 이유¶

    token, context window, prompt, completion, batch, sequence length를 설명하고, 같은 QPS라도 token shape가 다르면 완전히 다른 workload가 된다는 점을 다룬다.

    2부. GPU와 메모리 병목¶

    3장. GPU는 왜 빠르고 언제 느려지는가¶

    CUDA execution model, HBM/DRAM, compute-bound와 memory-bound 병목을 입문 수준으로 설명한다.

    4장. VRAM 예산: weights, KV cache, activations, overhead¶

    VRAM을 model weights, KV cache, activations, runtime overhead, fragmentation으로 분해해 rough sizing을 할 수 있게 한다.

    5장. KV cache가 LLM serving의 중심 병목이 되는 순간¶

    KV cache의 목적, 크기 공식, MHA/MQA/GQA/MLA/SWA 차이, PagedAttention, prefix caching, KV quantization을 하나의 mental model로 묶는다.

    3부. 단일 GPU에서 serving을 측정하기¶

    6장. 첫 LLM server: OpenAI-compatible endpoint 만들기¶

    model artifact, tokenizer, runtime, HTTP endpoint, client SDK의 관계를 설명하고 vLLM 기준의 첫 serving path를 구성한다.

    7장. TTFT, TPOT, throughput, p95를 읽는 법¶

    TTFT, TPOT, end-to-end latency, throughput, QPS, p95/p99, cost per token을 정의하고 benchmark 결과를 읽는 법을 다룬다.

    8장. Quantization은 언제 성능 최적화이고 언제 위험인가¶

    weight quantization, activation quantization, KV cache quantization을 분리하고 storage size, kernel support, accuracy, latency, throughput을 별도 축으로 본다.

    4부. Serving runtime 선택¶

    9장. vLLM: general-purpose LLM serving baseline¶

    vLLM의 scheduler, PagedAttention, continuous batching, prefix caching, metrics, distributed serving을 설명한다.

    10장. TensorRT-LLM, SGLang, llama.cpp는 무엇이 다른가¶

    TensorRT-LLM, SGLang, llama.cpp를 popularity가 아니라 workload, hardware, latency target, operational complexity 기준으로 비교한다.

    11장. Triton, KServe, Ray Serve: runtime 위의 platform 계층¶

    inference server, model serving platform, orchestration framework의 차이를 설명한다. KServe는 Kubernetes-native serving 예시로 다룬다.

    5부. Multi-GPU와 Kubernetes로 확장하기¶

    12장. Parallelism과 NCCL: 한 GPU를 넘을 때 생기는 문제¶

    tensor parallelism, pipeline parallelism, data parallelism, expert parallelism, NCCL collective communication을 설명한다.

    13장. Kubernetes GPU 운영: device plugin부터 rollout까지¶

    NVIDIA device plugin, GPU resource request, node labeling, image selection, model cache, readiness, autoscaling, canary rollout을 다룬다.

    6부. RAG, agents, application layer¶

    14장. RAG pipeline은 serving 문제가 아니라 release 문제다¶

    ingestion, chunking, embedding, index, retrieval, reranking, prompt assembly, citation, evaluation을 하나의 release artifact로 본다.

    15장. Agent workload는 왜 serving을 더 어렵게 만드는가¶

    tool call, long context, repeated prefix, retry, sandbox, memory, trace가 serving과 observability에 주는 영향을 설명한다.

    7부. Observability, evaluation, security¶

    16장. LLM observability: metrics, logs, traces, evals¶

    OpenTelemetry, Prometheus, Grafana 같은 system telemetry와 Langfuse, Phoenix 같은 LLM trace/evaluation 계층을 분리해 설명한다.

    17장. Evaluation과 benchmark: 숫자를 믿기 전에 조건을 기록하라¶

    workload shape, model/runtime version, hardware, concurrency, context length, prompt set, measurement method를 함께 기록하는 방법을 다룬다.

    18장. Security와 guardrails: prompt injection은 application boundary 문제다¶

    prompt injection, tool permission, RAG authorization, data exfiltration, audit logging, human approval boundary를 다룬다.

    8부. MLOps와 운영 로드맵¶

    19장. Model registry를 넘어 release record로¶

    model, tokenizer, adapter, prompt, retrieval index, serving runtime, quantization config, evaluator, safety policy를 release record로 묶는다.

    20장. 90일 학습 로드맵과 실습 순서¶

    단일 GPU 실험에서 Kubernetes deployment와 observability까지 이어지는 practical roadmap을 제시한다.
