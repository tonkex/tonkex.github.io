    샘플 챕터 계획¶

    추천 샘플 챕터는 5장. KV cache가 LLM serving의 중심 병목이 되는 순간 이다.

    현재 상태: 초안 v0.2 보강 완료. 공개 원고는 5장. KV cache 병목에 있다.

    이 장을 먼저 쓰는 이유¶

    KV cache는 이 책의 여러 주제를 한 번에 연결한다.

    • token과 context length
    • prefill과 decode
    • VRAM capacity
    • concurrent serving
    • PagedAttention
    • prefix caching
    • quantization
    • benchmark와 cost

    이 장이 잘 작동하면 책 전체의 설명 방식도 검증된다. 실무 문제에서 시작하고, 필요한 개념을 정의한 뒤, 도구 선택과 운영 지표로 돌아오는 구조를 만들 수 있기 때문이다.

    독자 질문¶

    KV cache는 정확히 무엇이고 왜 production serving에서 가장 먼저 봐야 하는가?

    장의 목표¶

    • KV cache가 왜 필요한지 설명한다.
    • KV cache 크기가 어떤 변수에 비례하는지 계산한다.
    • context length와 active sequence가 serving capacity를 어떻게 바꾸는지 보여준다.
    • PagedAttention, prefix caching, cache offload, KV quantization을 서로 다른 최적화 축으로 구분한다.
    • vLLM, SGLang, TensorRT-LLM을 비교할 때 KV cache 관점이 왜 필요한지 설명한다.

    초안 구성¶

    1. 사용자는 왜 첫 토큰을 기다리는가
    2. prefill과 decode의 차이
    3. KV cache의 정의
    4. KV cache 크기 공식
    5. model weights보다 KV cache가 더 큰 문제가 되는 경우
    6. MHA, MQA, GQA, MLA, sliding window attention
    7. PagedAttention이 해결한 문제
    8. prefix caching과 repeated prompt workload
    9. KV cache quantization의 이득과 위험
    10. 운영자가 확인해야 할 지표

    필요한 그림¶

    • prefill/decode timeline
    • KV cache lifecycle
    • KV cache size formula
    • Paged KV blocks
    • context length와 동시성에 따른 VRAM 사용량 변화
    • Q/K/V와 cache read path
    • prefix cache reuse

    작성 전 확인할 것¶

    • KV cache 계산 예시는 rough estimate로 표기하거나 공식 문서/논문 기준으로 검증한다.
    • aggressive KV quantization 관련 수치는 vLLM 문서나 원문 블로그를 다시 확인한 뒤 사용한다.
    • 개인 메모는 공개 본문에 직접 인용하지 않고, 원문 출처를 확인한 뒤 반영한다.

    초안 작성 결과¶

    • 정량 예시는 대표적인 decoder-only GQA 형태의 rough estimate로 명시했다.
    • PagedAttention, prefix caching, FP8 KV cache 관련 factual anchor는 공개 원문과 vLLM stable docs를 기준으로 확인했다.
    • 공개 원고에는 개인 자료 경로와 내부 메모명을 노출하지 않았다.
    • v0.2에서 Q/K/V 흐름, active token 예산, scheduler pressure, fragmentation, prefix caching 실패 원인, 운영 정책, 장애 triage, 실습 표를 추가했다.
    • v0.2에서 그림을 2개에서 6개로 늘렸다.
