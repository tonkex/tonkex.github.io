    11мһҘ. Triton, KServe, Ray Serve: runtime мң„мқҳ platform кі„мёө¶

    Draft v0.1 Updated: 2026-07-10 Part 4 - Serving runtime м„ нғқ

    10мһҘм—җм„ң мҡ°лҰ¬лҠ” vLLM, TensorRT-LLM, SGLang, llama.cppлҘј runtimeмңјлЎң л№„көҗн–ҲлӢӨ. к·ёлҹ°лҚ° productionм—җ к°Җк№ҢмӣҢм§ҲмҲҳлЎқ м§Ҳл¬ёмқҙ л°”лҖҗлӢӨ.

    лӘЁлҚё м„ңлІ„лҘј лқ„мҡҙ лӢӨмқҢ production platformмқҖ л¬ҙм—Үмқ„ лҚ” н•ҙмӨҳм•ј н•ҳлҠ”к°Җ?

    RuntimeмқҖ лӘЁлҚёмқ„ мӢӨн–үн•ңлӢӨ. PlatformмқҖ к·ё runtimeмқ„ м•Ҳм „н•ҳкІҢ л°°нҸ¬н•ҳкі , лқјмҡ°нҢ…н•ҳкі , мҠӨмјҖмқјн•ҳкі , кҙҖмёЎн•ҳкі , лҗҳлҸҢлҰҙ мҲҳ мһҲкІҢ л§Ңл“ лӢӨ. мқҙ л‘ҳмқ„ м„һмңјл©ҙ YAMLмқҖ л§Һм•„м§ҖлҠ”лҚ° мҡҙмҳҒ мұ…мһ„мқҖ нқҗл Өм§„лӢӨ.

    мқҙ мһҘмқҳ лҸ…мһҗ м§Ҳл¬ёмқҖ лӢӨмқҢкіј к°ҷлӢӨ.

    Inference server, model serving platform, orchestration frameworkлҠ” м–ҙл–»кІҢ лӢӨлҘҙкі , Triton, KServe, Ray ServeлҠ” к°Ғк°Ғ м–ҙл””м—җ лҶ“м•„м•ј н•ҳлҠ”к°Җ?

    н•өмӢ¬мқҖ н•ҳлӮҳлӢӨ. PlatformмқҖ "лӘЁлҚёмқ„ мӢӨн–үн•ҳлҠ” л°©лІ•"ліҙлӢӨ "лӘЁлҚё м„ңл№„мҠӨлҘј мҡҙмҳҒ к°ҖлҠҘн•ң лӢЁмң„лЎң л§Ңл“ңлҠ” л°©лІ•"м—җ к°Җк№қлӢӨ.

    runtimeкіј platformмқ„ лЁјм Җ л¶„лҰ¬н•ңлӢӨ¶

    10мһҘмқҳ runtime м„ нғқмқҖ model artifactмҷҖ GPU execution pathм—җ к°Җк№Ңмӣ лӢӨ. 11мһҘмқҳ platform м„ нғқмқҖ мҡҙмҳҒ мұ…мһ„м—җ к°Җк№қлӢӨ.

    кі„мёө мЈјлҗң м§Ҳл¬ё мҳҲ
    runtime мқҙ лӘЁлҚёмқ„ м–ҙл–Ө engineмңјлЎң мӢӨн–үн•  кІғмқёк°Җ? vLLM, TensorRT-LLM, SGLang, llama.cpp
    inference server м—¬лҹ¬ model/backendлҘј м–ҙл–Ө protocolкіј schedulerлЎң м„ңл№ҷн•  кІғмқёк°Җ? Triton Inference Server
    serving platform Kubernetesм—җм„ң model service lifecycleмқ„ м–ҙл–»кІҢ н‘ңмӨҖнҷ”н•  кІғмқёк°Җ? KServe
    orchestration framework Python business logic, model composition, distributed executionмқ„ м–ҙл–»кІҢ л¬¶мқ„ кІғмқёк°Җ? Ray Serve
    gateway / product layer auth, quota, tenant, routing, fallback, auditлҘј м–ҙл–»кІҢ м Ғмҡ©н•  кІғмқёк°Җ? API gateway, service mesh, internal platform

    мқҙ кө¬л¶„мқҙ мӨ‘мҡ”н•ң мқҙмң лҠ” к°„лӢЁн•ҳлӢӨ. Runtimeмқ„ л°”кҝ”лҸ„ rollout, autoscaling, auth, model registry, incident responseк°Җ мһҗлҸҷмңјлЎң мғқкё°м§Җ м•ҠлҠ”лӢӨ. л°ҳлҢҖлЎң platformмқ„ лҸ„мһ…н•ҙлҸ„ runtimeмқҳ KV cache лі‘лӘ©мқҙлӮҳ tokenizer mismatchк°Җ мӮ¬лқјм§Җм§ҖлҠ” м•ҠлҠ”лӢӨ.

    Triton: multi-framework inference server¶

    NVIDIA Triton Inference ServerлҠ” м—¬лҹ¬ deep learningкіј machine learning frameworkмқҳ modelмқ„ л°°нҸ¬н•ҳкё° мң„н•ң open-source inference serving softwareлӢӨ. кіөмӢқ л¬ём„ңлҠ” TensorRT, PyTorch, ONNX, OpenVINO, Python backend л“ұ м—¬лҹ¬ frameworkлҘј м§Җмӣҗн•ҳкі , cloud, data center, edge, embedded нҷҳкІҪм—җм„ң inferenceлҘј м ңкіөн•ңлӢӨкі  м„ӨлӘ…н•ңлӢӨ.

    Tritonмқҳ мӨ‘мӢ¬ к°ңл…җмқҖ "model repositoryмҷҖ backendлҘј к°Җм§„ inference server"лӢӨ. RequestлҠ” HTTP/REST лҳҗлҠ” gRPCлЎң л“Өм–ҙмҳӨкі , modelлі„ schedulerлЎң лқјмҡ°нҢ…лҗңлӢӨ. SchedulerлҠ” н•„мҡ”н•ҳл©ҙ batchingмқ„ мҲҳн–үн•ҳкі , backendк°Җ мӢӨм ң inferenceлҘј мӢӨн–үн•ңлӢӨ.

    Tritonмқҙ мһҳ л§һлҠ” мғҒнҷ©мқҖ лӢӨмқҢкіј к°ҷлӢӨ.

    • LLMлҝҗ м•„лӢҲлқј vision, ranking, embedding, tabular model л“ұ м—¬лҹ¬ framework modelмқ„ н•Ёк»ҳ м„ңл№ҷн•ңлӢӨ.
    • HTTP/gRPC inference protocol, model repository, model management APIк°Җ н•„мҡ”н•ҳлӢӨ.
    • dynamic batching, sequence batching, concurrent model execution, ensemble pipelineмқҙ мӨ‘мҡ”н•ҳлӢӨ.
    • TensorRT-LLM, vLLM backend, Python backend л“ұ backend кё°л°ҳмңјлЎң model servingмқ„ н‘ңмӨҖнҷ”н•ҳкі  мӢ¶лӢӨ.
    • KubernetesмҷҖ л¶ҷмқј readiness/liveness, metrics, tracing, model analyzer к°ҷмқҖ мҡҙмҳҒ лҸ„кө¬к°Җ н•„мҡ”н•ҳлӢӨ.

    TritonмқҖ runtimeмқҙкё°лҸ„ н•ҳкі  serverмқҙкё°лҸ„ н•ҳм§Җл§Ң, Kubernetes-native platform м „мІҙлҠ” м•„лӢҲлӢӨ. Tritonмқҙ model executionкіј server protocolмқ„ мһҳ лӢӨлЈ¬лӢӨл©ҙ, Kubernetes л°°нҸ¬, rollout, traffic split, autoscaling policy, tenant policyлҠ” лӢӨлҘё кі„мёөм—җм„ң лӢӨлӨ„м•ј н•ңлӢӨ.

    Tritonмқҙ н•ҙмЈјлҠ” кІғ лі„лҸ„лЎң м„Өкі„н•ҙм•ј н•ҳлҠ” кІғ
    model repository, backend, scheduler, batching service-level canary rollout
    HTTP/gRPC/KServe protocol endpoint Kubernetes CRD lifecycle
    readiness/liveness, metrics, tracing tenant auth, quota, product gateway
    ensemble/BLS pipeline application-level eval, release approval

    AI мқён”„лқј м—”м§ҖлӢҲм–ҙлҠ” Tritonмқ„ "NVIDIAмҡ© л№ лҘё м„ңлІ„" м •лҸ„лЎңл§Ң ліҙл©ҙ м•Ҳ лҗңлӢӨ. Tritonмқҳ к°•м җмқҖ multi-framework inference serverлЎңм„ң model managementмҷҖ protocol, batching, backend нҷ•мһҘм„ұмқ„ м ңкіөн•ҳлҠ” лҚ° мһҲлӢӨ.

    KServe: Kubernetes-native model serving platform¶

    KServeлҠ” Kubernetes мң„м—җм„ң predictive AIмҷҖ generative AI inferenceлҘј н‘ңмӨҖнҷ”н•ҳлҠ” serving platformмқҙлӢӨ. кіөмӢқ мӮ¬мқҙнҠёлҠ” KServeлҘј scalable, multi-framework deploymentлҘј мң„н•ң Kubernetes кё°л°ҳ platformмңјлЎң м„ӨлӘ…н•ҳкі , InferenceService CRDк°Җ autoscaling, networking, health checking, server configuration ліөмһЎлҸ„лҘј мәЎмҠҗнҷ”н•ңлӢӨкі  м„ӨлӘ…н•ңлӢӨ.

    KServeмқҳ мӨ‘мӢ¬ лӢЁмң„лҠ” InferenceServiceлӢӨ. мӮ¬мҡ©мһҗлҠ” model format, storage URI, resource request, GPU limit к°ҷмқҖ м •ліҙлҘј м„ м–ён•ҳкі , KServeлҠ” deployment lifecycleкіј routingмқ„ кҙҖлҰ¬н•ңлӢӨ.

    KServeк°Җ мһҳ л§һлҠ” мғҒнҷ©мқҖ лӢӨмқҢкіј к°ҷлӢӨ.

    • нҢҖмқҙ мқҙлҜё KubernetesлҘј production platformмңјлЎң м“°кі  мһҲлӢӨ.
    • лӘЁлҚё л°°нҸ¬ лӢЁмң„лҘј CRDлЎң н‘ңмӨҖнҷ”н•ҳкі  мӢ¶лӢӨ.
    • model serviceл§ҲлӢӨ networking, autoscaling, health check, revision, traffic routingмқ„ мқјкҙҖлҗҳкІҢ кҙҖлҰ¬н•ҙм•ј н•ңлӢӨ.
    • predictive modelкіј generative modelмқ„ к°ҷмқҖ platform кҙҖм җм—җм„ң мҡҙмҳҒн•ҳкі  мӢ¶лӢӨ.
    • canary rollout, A/B testing, inference graph, pre/post processing к°ҷмқҖ deployment patternмқҙ мӨ‘мҡ”н•ҳлӢӨ.

    KServeмқҳ мһҘм җмқҖ runtimeмқ„ к°җмӢёлҠ” platform contractлҘј м ңкіөн•ңлӢӨлҠ” м җмқҙлӢӨ. мҳҲлҘј л“Өм–ҙ LLM servingм—җм„ңлҠ” vLLM лҳҗлҠ” llm-d к°ҷмқҖ backendлҘј нҷңмҡ©н•ҳкі , OpenAI-compatible protocol, model caching, GPU acceleration, autoscaling к°ҷмқҖ platform кё°лҠҘмқ„ н•Ёк»ҳ м ңкіөн•  мҲҳ мһҲлӢӨ.

    н•ҳм§Җл§Ң KServeлҸ„ лӘЁл“  л¬ём ңлҘј н•ҙкІ°н•ҳм§Җ м•ҠлҠ”лӢӨ.

    KServeк°Җ н•ҙмЈјлҠ” кІғ м—¬м „нһҲ н•„мҡ”н•ң кІғ
    Kubernetes CRD кё°л°ҳ model lifecycle model н’Ҳм§Ҳ evalкіј release approval
    autoscaling, networking, health checking GPU capacity planningкіј node pool м„Өкі„
    canary rollout, A/B testing application-level fallback policy
    model storage URIмҷҖ resource declaration model artifact versioning кё°мӨҖ
    predictive/generative serving abstraction runtimeлі„ tuningкіј metric н•ҙм„қ

    KServeлҘј YAMLл¶Җн„° л°°мҡ°л©ҙ лӘ©м Ғмқ„ лҶ“м№ҳкё° мүҪлӢӨ. мӨ‘мҡ”н•ң кІғмқҖ InferenceService specмқҳ лӘЁл“  н•„л“ңлҘј мҷёмҡ°лҠ” кІғмқҙ м•„лӢҲлқј, model servingмқ„ Kubernetes production objectлЎң л°”кҫёлҠ” мұ…мһ„мқҙ л¬ҙм—Үмқём§Җ мқҙн•ҙн•ҳлҠ” кІғмқҙлӢӨ.

    Ray Serve: Python compositionкіј distributed serving¶

    Ray ServeлҠ” Ray мң„м—җм„ң online inference APIлҘј л§Ңл“Өкё° мң„н•ң scalable model serving libraryлӢӨ. кіөмӢқ л¬ём„ңлҠ” Ray Serveк°Җ framework-agnosticмқҙл©° PyTorch, TensorFlow, Keras, Scikit-Learn modelл¶Җн„° arbitrary Python business logicк№Ңм§Җ н•ҳлӮҳмқҳ toolkitмңјлЎң м„ңл№ҷн•  мҲҳ мһҲлӢӨкі  м„ӨлӘ…н•ңлӢӨ. лҳҗн•ң LLM servingмқ„ мң„н•ң response streaming, dynamic request batching, multi-node/multi-GPU serving к°ҷмқҖ кё°лҠҘмқ„ м ңкіөн•ңлӢӨ.

    Ray Serveк°Җ нҠ№нһҲ к°•н•ң м§Җм җмқҖ model compositionмқҙлӢӨ. м—¬лҹ¬ model, retriever, preprocessor, postprocessor, validation logic, business ruleмқ„ Python codeлЎң л¬¶кі , к°Ғ stepмқ„ лҸ…лҰҪм ҒмңјлЎң мҠӨмјҖмқјн•  мҲҳ мһҲлӢӨ.

    Ray Serveк°Җ мһҳ л§һлҠ” мғҒнҷ©мқҖ лӢӨмқҢкіј к°ҷлӢӨ.

    • inference pathк°Җ лӢЁмқј model callмқҙ м•„лӢҲлқј Python business logicкіј м—¬лҹ¬ componentмқҳ мЎ°н•©мқҙлӢӨ.
    • RAG, agent, validation, custom routing, postprocessingмқ„ н•ң application graphлЎң лӢӨлЈЁкі  мӢ¶лӢӨ.
    • Kubernetes м—Ҷмқҙ лЎңм»¬ лҳҗлҠ” Ray clusterм—җм„ң лЁјм Җ к°ңл°ңн•ҳкі , лӮҳмӨ‘м—җ KubeRayлЎң нҷ•мһҘн•ҳкі  мӢ¶лӢӨ.
    • к°Ғ componentлҘј лҸ…лҰҪм ҒмңјлЎң autoscaleн•ҳкі  мӢ¶лӢӨ.
    • model-specific optimized serverліҙлӢӨ application compositionкіј distributed executionмқҙ лҚ” мӨ‘мҡ”н•ҳлӢӨ.

    Ray ServeлҠ” KServeлӮҳ Tritonкіј лӢӨлҘё м§Ҳл¬ём—җ лӢөн•ңлӢӨ. "мқҙ лӘЁлҚёмқ„ м–ҙл–Ө backendлЎң м„ңл№ҷн• к№Ң"ліҙлӢӨ "м—¬лҹ¬ modelкіј Python logicмңјлЎң лҗң inference applicationмқ„ м–ҙл–»кІҢ scalable serviceлЎң л§Ңл“Өк№Ң"м—җ к°Җк№қлӢӨ.

    мЈјмқҳн•  м җлҸ„ мһҲлӢӨ. Ray Serve л¬ём„ңлҸ„ Ray Serveк°Җ full-fledged ML platformмқҖ м•„лӢҲлқјкі  м„ӨлӘ…н•ңлӢӨ. Model lifecycle кҙҖлҰ¬, м„ұлҠҘ мӢңк°Ғнҷ”, organization-wide governanceлҠ” лі„лҸ„ platform кі„мёөмқҙ н•„мҡ”н•  мҲҳ мһҲлӢӨ. лҳҗн•ң model-specific kernel optimizationмқҖ TritonмқҙлӮҳ TensorRT-LLM к°ҷмқҖ runtime/server кі„мёөкіј лі„лҸ„лЎң лҙҗм•ј н•ңлӢӨ.

    м„ё лҸ„кө¬лҘј к°ҷмқҖ н‘ңм—җ м–өм§ҖлЎң л„Јм§Җ м•ҠлҠ”лӢӨ¶

    Triton, KServe, Ray ServeлҠ” лӘЁл‘җ "serving"мқҙлқјлҠ” лӢЁм–ҙлҘј м“°м§Җл§Ң к°ҷмқҖ лІ”мЈјмқҳ м ңн’Ҳмқҙ м•„лӢҲлӢӨ.

    лҸ„кө¬ к°ҖмһҘ к°Җк№Ңмҡҙ м—ӯн•  лЁјм Җ л¬јм–ҙліј м§Ҳл¬ё
    Triton multi-framework inference server м—¬лҹ¬ backendмҷҖ modelмқ„ protocol, batching, metricsлЎң н‘ңмӨҖнҷ”н•  кІғмқёк°Җ?
    KServe Kubernetes-native model serving platform model service lifecycle, rollout, autoscalingмқ„ Kubernetes CRDлЎң н‘ңмӨҖнҷ”н•  кІғмқёк°Җ?
    Ray Serve programmable distributed serving framework model compositionкіј Python business logicмқ„ scalable serviceлЎң л§Ңл“Ө кІғмқёк°Җ?

    л”°лқјм„ң м„ нғқмқҖ л°°нғҖм Ғмқҙм§Җ м•Ҡмқ„ мҲҳ мһҲлӢӨ. м–ҙл–Ө мЎ°м§ҒмқҖ Tritonмқ„ model serverлЎң м“°кі  KServeлЎң Kubernetes lifecycleмқ„ кҙҖлҰ¬н•  мҲҳ мһҲлӢӨ. м–ҙл–Ө нҢҖмқҖ Ray ServeлЎң RAG application graphлҘј кө¬м„ұн•ҳкі , лӮҙл¶Җ stepм—җм„ң optimized runtimeмқ„ нҳём¶ңн•  мҲҳ мһҲлӢӨ. мӨ‘мҡ”н•ң кІғмқҖ "н•ҳлӮҳл§Ң кіЁлқјм•ј н•ңлӢӨ"к°Җ м•„лӢҲлқј "м–ҙл–Ө кі„мёөмқҳ мұ…мһ„мқ„ н•ҙкІ°н•ҳлҠ”к°Җ"лӢӨ.

    Canary rolloutмқҖ platform мұ…мһ„мқ„ ліҙм—¬ мӨҖлӢӨ¶

    Runtime кҙҖм җм—җм„ңлҠ” мғҲ model artifactлҘј лЎңл“ңн•ҳл©ҙ лҒқмІҳлҹј ліҙмқёлӢӨ. Platform кҙҖм җм—җм„ңлҠ” к·ёл•Ңл¶Җн„°к°Җ мӢңмһ‘мқҙлӢӨ. Production rolloutмқҖ ліҙнҶө лӢӨмқҢ м§Ҳл¬ёмқ„ нҸ¬н•Ён•ңлӢӨ.

    • мғҲ model/runtime versionмқ„ лӘҮ нҚјм„јнҠё trafficм—җ ліҙлӮј кІғмқёк°Җ?
    • p95 latency, error rate, refusal rate, tool call success rateк°Җ лӮҳл№ м§Җл©ҙ мһҗлҸҷмңјлЎң л©Ҳм¶ң кІғмқёк°Җ?
    • кё°мЎҙ versionмңјлЎң rollbackн•  кІҪлЎңк°Җ мһҲлҠ”к°Җ?
    • user cohort, tenant, endpointлі„лЎң trafficмқ„ лӮҳлҲҢ мҲҳ мһҲлҠ”к°Җ?
    • metricкіј traceк°Җ version labelмқ„ нҸ¬н•Ён•ҳлҠ”к°Җ?

    KServeлҠ” canary rolloutкіј A/B testingмқ„ platform кё°лҠҘмңјлЎң к°•мЎ°н•ңлӢӨ. Ray ServeлҠ” application graphмҷҖ component-level autoscalingмқ„ нҶөн•ҙ rollout м „лһөмқ„ л§Ңл“Ө мҲҳ мһҲлӢӨ. TritonмқҖ model repositoryмҷҖ model management, metricsлҘј м ңкіөн•ҳм§Җл§Ң service-level traffic splitмқҖ Kubernetes/service mesh/gateway кі„мёөкіј н•Ёк»ҳ м„Өкі„н•ҙм•ј н•ңлӢӨ.

    Platformмқ„ лҸ„мһ…н•ҳкё° м „м—җ кі м •н•  кі„м•Ҫ¶

    Platformмқ„ кі лҘҙкё° м „м—җ лЁјм Җ service contractлҘј л¬ём„ңнҷ”н•ҙм•ј н•ңлӢӨ.

    кі„м•Ҫ м§Ҳл¬ё
    endpoint contract OpenAI-compatibleмқёк°Җ, native protocolмқёк°Җ, gRPCмқёк°Җ?
    artifact contract HF checkpoint, TensorRT engine, GGUF, container image, model storage URI мӨ‘ л¬ҙм—Үмқҙ release лӢЁмң„мқёк°Җ?
    resource contract GPU type, GPU count, CPU, memory, model cache, local SSDк°Җ м–јл§ҲлӮҳ н•„мҡ”н•ңк°Җ?
    scaling contract л¬ҙм—Үмқ„ кё°мӨҖмңјлЎң replicaлҘј лҠҳлҰҙ кІғмқёк°Җ? QPS, concurrency, queue, GPU memory, custom metric?
    rollout contract canary traffic л№„мңЁ, promotion gate, rollback triggerлҠ” л¬ҙм—Үмқёк°Җ?
    observability contract metric, log, trace, eval resultм—җ model/runtime/version labelмқҙ мһҲлҠ”к°Җ?
    failure contract timeout, OOM, bad output, safety failure, dependency failureлҘј м–ҙл–»кІҢ кө¬л¶„н•ҳлҠ”к°Җ?

    мқҙ кі„м•Ҫ м—Ҷмқҙ platformл¶Җн„° лҸ„мһ…н•ҳл©ҙ YAMLкіј controllerлҠ” мғқкё°м§Җл§Ң мҡҙмҳҒ нҢҗлӢЁмқҖ м—¬м „нһҲ мӮ¬лһҢ лЁёлҰҝмҶҚм—җ лӮЁлҠ”лӢӨ.

    мӢӨмҠө: platform responsibility checklist л§Ңл“Өкё°¶

    10мһҘм—җм„ң runtime нӣ„ліҙлҘј кі лҘё л’Ө, 11мһҘм—җм„ңлҠ” лӢӨмқҢ checklistлҘј мұ„мҡҙлӢӨ.

    н•ӯлӘ© нҳ„мһ¬ мғҒнғң н•„мҡ”н•ң platform мұ…мһ„ нӣ„ліҙ
    runtime vLLM / TensorRT-LLM / SGLang / llama.cpp мӢӨн–үкіј tuning runtime
    packaging container image / model storage URI мһ¬нҳ„ к°ҖлҠҘн•ң release CI/CD, registry
    deployment manual command / Kubernetes YAML rolloutкіј rollback KServe, Ray Serve, custom

    мқҙ н‘ңмқҳ лӘ©м ҒмқҖ нҠ№м • лҸ„кө¬лҘј кі лҘҙлҠ” кІғмқҙ м•„лӢҲлқј л№ м§„ мұ…мһ„мқ„ м°ҫлҠ” кІғмқҙлӢӨ. мҳҲлҘј л“Өм–ҙ runtime metricsлҠ” мһҲлҠ”лҚ° version-aware dashboardк°Җ м—ҶлӢӨл©ҙ, canary rolloutмқ„ н•ҙлҸ„ м–ҙл–Ө versionмқҙ л¬ём ңмқём§Җ м¶”м Ғн•ҳкё° м–ҙл өлӢӨ. Kubernetes deploymentлҠ” мһҲлҠ”лҚ° model artifact release recordк°Җ м—ҶлӢӨл©ҙ rollbackмқҖ нҳ•мӢқл§Ң мһҲлҠ” м…ҲмқҙлӢӨ.

    мӢӨнҢЁ нҢЁн„ҙ¶

    YAMLмқ„ platformмқҙлқјкі  м°©к°Ғн•ңлӢӨ¶

    Kubernetes YAMLмқҙ мһҲлӢӨкі  platformмқҙ лҗҳлҠ” кІғмқҖ м•„лӢҲлӢӨ. Release approval, traffic policy, metric gate, rollback runbookмқҙ м—ҶлӢӨл©ҙ л°°нҸ¬ нҢҢмқјл§Ң мһҲлҠ” мғҒнғңлӢӨ.

    runtime tuningкіј platform scalingмқ„ м„һлҠ”лӢӨ¶

    max_num_seqs, KV cache size, batch token limitмқҖ runtime tuningмқҙлӢӨ. Replica count, node pool, autoscaling policyлҠ” platform scalingмқҙлӢӨ. л‘ҳмқҖ м—°кІ°лҗҳм§Җл§Ң к°ҷмқҖ л ҲлІ„к°Җ м•„лӢҲлӢӨ.

    GPU autoscalingмқ„ QPSл§ҢмңјлЎң н•ңлӢӨ¶

    LLM servingм—җм„ң QPS 10мқҖ token shapeм—җ л”°лқј мҷ„м „нһҲ лӢӨлҘё л¶Җн•ҳлӢӨ. Platform scalingмқҖ QPSлҝҗ м•„лӢҲлқј input token, output token, active sequence, queue time, KV cache usageлҘј н•Ёк»ҳ лҙҗм•ј н•ңлӢӨ.

    canaryлҘј latencyл§ҢмңјлЎң нҢҗлӢЁн•ңлӢӨ¶

    мғҲ modelмқҙ л№ лҘҙлҚ”лқјлҸ„ tool call format, grounding, refusal behavior, citation qualityк°Җ лӮҳл№ м§Ҳ мҲҳ мһҲлӢӨ. Generative AI rolloutмқҖ latency gateмҷҖ quality gateк°Җ н•Ёк»ҳ н•„мҡ”н•ҳлӢӨ.

    platformмқҙ securityмҷҖ governanceлҘј мһҗлҸҷмңјлЎң н•ҙкІ°н•ңлӢӨкі  ліёлӢӨ¶

    KServeлӮҳ Ray ServeлҘј лҸ„мһ…н•ҙлҸ„ tenant isolation, auth, audit log, data retention, policy enforcementлҠ” лі„лҸ„ м„Өкі„к°Җ н•„мҡ”н•ҳлӢӨ.

    м •лҰ¬¶

    • RuntimeмқҖ model execution pathлҘј лӢӨлЈЁкі , platformмқҖ deployment lifecycleкіј мҡҙмҳҒ мұ…мһ„мқ„ лӢӨлЈ¬лӢӨ.
    • TritonмқҖ multi-framework inference serverлЎң model repository, backend, scheduler, batching, metricsм—җ к°•н•ҳлӢӨ.

    • Ray ServeлҠ” Python business logicкіј model compositionмқ„ scalable online serviceлЎң л§Ңл“ңлҠ” лҚ° к°•н•ҳлӢӨ.
    • Canary rolloutмқҖ platform мұ…мһ„мқ„ к°ҖмһҘ мһҳ ліҙм—¬ мЈјлҠ” мҳҲлӢӨ. Traffic split, metric gate, eval gate, rollback pathк°Җ н•Ёк»ҳ мһҲм–ҙм•ј н•ңлӢӨ.
    • Platform лҸ„мһ… м „м—җлҠ” endpoint, artifact, resource, scaling, rollout, observability, failure contractлҘј лЁјм Җ л¬ём„ңнҷ”н•ҙм•ј н•ңлӢӨ.
